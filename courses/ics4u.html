1.1 — Variables, Data Types, and Expressions
Primitive types (int, float, str, bool); variable assignment; type conversion; operator precedence; basic expression evaluation
# Variable assignment and type conversion name = "Ada" age = 17 gpa = 3.85 is_senior = True # Build a record string with explicit conversions record = name + " (" + str(age) + "): " + str(gpa) print(record) # Ada (17): 3.85
📖 Lesson Notes — Ontario Curriculum

Every value a program manipulates has a data type, and every type carries with it a set of legal operations. Choosing the right primitive type is the first decision a programmer makes because it controls memory layout, the range of representable values, and which operators are valid. Confusing types is the source of a surprising number of bugs: "7" + "8" produces the string "78" in Python and Java, but 7 + 8 produces the integer 15. Grade 12 students must learn to reason about types before they reason about logic.

Python uses dynamic typing: a variable name is a label that can be rebound to any object. By contrast, Java uses static typing, where the compiler verifies the type at every assignment. In both languages, the value stored in memory has a fixed type at any given moment. Type conversion (also called casting) is the act of producing a new value of a different type — for example, str(age) creates a string version of the integer age without altering the original. Implicit conversion (3 + 4.0 becoming 7.0) is called type coercion and is the most common source of silent floating-point error.

Expressions are built from operators and operands, and the order in which they evaluate is governed by operator precedence. Multiplication binds tighter than addition; exponentiation in Python (**) binds tighter than unary minus on the left but not on the right; comparison binds looser than arithmetic. When in doubt, add parentheses — they cost nothing at runtime and prevent precedence bugs.

  • Primitive type — a built-in type that is not a collection (int, float, bool, str).
  • Literal — a value written directly in source code, e.g. 17 or "Ada".
  • Identifier — the name of a variable, function, or class.
  • Casting / type conversion — explicit conversion from one type to another, e.g. int("12").
  • Operator precedence — rules that determine the order in which sub-expressions evaluate.
  • Floating-point error — the rounding caused by binary representation of decimal fractions.

Worked example — precedence and floating-point

>>> 2 + 3 * 4 ** 2     # ** first → 16, then * → 48, then + → 50
50
>>> (2 + 3) * 4 ** 2   # () forces 5 first, ** still binds tighter than *
80
>>> 0.1 + 0.2          # binary cannot represent 0.1 exactly
0.30000000000000004
>>> round(0.1 + 0.2, 2)
0.3

The exponent 4 ** 2 evaluates first because ** has the highest precedence among arithmetic operators. Floating-point arithmetic in IEEE-754 cannot represent 0.1 exactly, so equality comparisons on floats should use a tolerance (abs(a - b) < 1e-9) rather than ==. Each operation here is O(1), so the running time of the expression is constant regardless of operand magnitude.

  • Forgetting to convert input() from string to number: input() + 1 raises TypeError.
  • Comparing floats with == — always use a tolerance.
  • Using = when you mean ==, or vice versa.
  • Assuming integer division: in Python 3, 5 / 2 is 2.5, not 2. Use // for floor division.
  • Overwriting a built-in name like list or str with your own variable.

1.2 — Input, Output, and String Formatting
input(), print(), f-strings, escape sequences, integer vs. float input parsing
# Build an interactive area calculator length = float(input("Length (m): ")) width = float(input("Width (m): ")) area = length * width print(f"Area = {area:.2f} m²")
📖 Lesson Notes — Ontario Curriculum

A program that cannot communicate with the outside world is not very useful. Input and output (I/O) are the bridges between the program's internal state and the user or another system. In Python, input() always returns a str, regardless of what the user types — even "42" arrives as four characters, not a number. The programmer must explicitly parse it with int() or float() before performing arithmetic. Forgetting this conversion is the single most common bug in introductory programs.

Output formatting matters because raw values rarely look the way humans want to read them. A bank balance of 1234.5 should display as $1,234.50 with two decimal places and a thousands separator. Python's f-strings (formatted string literals, introduced in Python 3.6) embed expressions directly inside string literals with a format specifier after a colon: f"{value:,.2f}" means "thousands separator, two decimal places, fixed-point notation". F-strings are faster, easier to read, and less error-prone than the older % operator or the .format() method.

Escape sequences let strings contain characters that would otherwise have special meaning. \n produces a newline, \t produces a tab, \\ produces a literal backslash, and \" escapes a double-quote inside a double-quoted string. Raw strings, prefixed with r, disable escapes — they are handy for Windows file paths and regular expressions.

  • Standard input (stdin) — the default source of input, usually the keyboard.
  • Standard output (stdout) — the default destination of output, usually the terminal.
  • f-string — an expression-embedded literal of the form f"…{expr:spec}…".
  • Format specifier — the part after the colon (.2f, >10, 05d) that controls width, precision, alignment.
  • Escape sequence — a two-character code beginning with \ that represents a non-printable character.
  • Concatenation — joining strings end-to-end with +.

Worked example — receipt formatting

item  = input("Item: ")          # str
qty   = int(input("Qty: "))      # int — parse!
price = float(input("Price: "))  # float — parse!

subtotal = qty * price
tax      = subtotal * 0.13       # Ontario HST
total    = subtotal + tax

print(f"{'Item':<15}{'Qty':>5}{'Price':>10}")
print(f"{item:<15}{qty:>5}{price:>10.2f}")
print(f"{'Subtotal':<20}{subtotal:>10.2f}")
print(f"{'HST (13%)':<20}{tax:>10.2f}")
print(f"{'TOTAL':<20}{total:>10.2f}")

The <15 specifier left-aligns to width 15; >10.2f right-aligns a float in width 10 with two decimals. Each print is an O(1) I/O operation. The arithmetic itself is O(1); the runtime is dominated by I/O latency, not computation.

  • Forgetting to wrap input() with int() or float() — leads to TypeError when arithmetic is attempted.
  • Calling int("3.5") — raises ValueError. Use float() first, then int() if truncation is wanted.
  • Using print(x, y) (which auto-inserts a space) when concatenation was intended.
  • Forgetting the f prefix: "{x}" is the literal string {x}, not the value of x.
  • Mixing tabs and spaces in formatted output — terminals may render them differently.

1.3 — Selection: if/elif/else and Boolean Logic
Comparison operators, logical and/or/not, truth tables, short-circuit evaluation, the dangling-else problem
# Letter-grade classifier (Ontario Achievement Chart) def letter_grade(percent): if percent >= 80: return "Level 4" elif percent >= 70: return "Level 3" elif percent >= 60: return "Level 2" elif percent >= 50: return "Level 1" else: return "R"
📖 Lesson Notes — Ontario Curriculum

Selection — choosing which branch of code to run — is one of the three pillars of structured programming alongside sequence and iteration. The Boolean expression in an if evaluates to either True or False, and the interpreter takes the corresponding branch. Cascaded if/elif/else chains are mutually exclusive: at most one branch executes, and they are evaluated top-to-bottom. The order of conditions is therefore semantically meaningful, not just stylistic. Writing the most restrictive condition first is a frequent source of bugs (e.g., if percent >= 50 before if percent >= 80 would classify every passing grade as Level 1).

Boolean logic is built from the three connectives and, or, and not. A truth table enumerates all input combinations. Python (and Java) use short-circuit evaluation: in A and B, if A is false, B is never evaluated; in A or B, if A is true, B is never evaluated. This is more than an optimisation — it lets programmers write guards like if x != 0 and 1/x > 0.5 without raising a division-by-zero exception when x is zero.

De Morgan's laws are essential for refactoring conditionals: not (A and B) is equivalent to (not A) or (not B), and not (A or B) is equivalent to (not A) and (not B). Students should be able to apply these mechanically to simplify nested negations.

  • Boolean expression — any expression that evaluates to True or False.
  • Comparison operator — ==, !=, <, <=, >, >=.
  • Logical operator — and, or, not (in Java: &&, ||, !).
  • Short-circuit evaluation — second operand is skipped when the first determines the result.
  • Truthy / Falsy — non-Boolean values evaluated in a Boolean context (0, "", [], None are falsy).
  • Dangling-else — ambiguity in nested ifs without braces, resolved in Python by indentation.

Worked example — input validation with short-circuit logic

def safe_divide(a_str, b_str):
    if not a_str.lstrip('-').isdigit() or not b_str.lstrip('-').isdigit():
        return "Inputs must be integers"
    a, b = int(a_str), int(b_str)
    if b == 0:
        return "Cannot divide by zero"
    return a / b

print(safe_divide("12", "3"))   # 4.0
print(safe_divide("12", "0"))   # Cannot divide by zero
print(safe_divide("12", "x"))   # Inputs must be integers

The or in the first check short-circuits: if a_str is non-numeric, b_str is never tested. This protects against the ValueError that would arise from int("x"). The whole function runs in O(n) where n is the length of the input string (because of isdigit); the arithmetic itself is O(1).

  • Writing if x = 5 instead of if x == 5 (a syntax error in Python, a logic bug in C/Java).
  • Putting the most general condition first in an if/elif chain, making later branches unreachable.
  • Chaining comparisons in Java the way Python allows: 1 < x < 10 works in Python; in Java write 1 < x && x < 10.
  • Forgetting that and/or return one of the operands, not necessarily True/False: 0 or "hello" returns "hello".
  • Negating a compound condition without De Morgan's laws — easy to invert the wrong operator.

1.4 — Iteration: for, while, and Loop Tracing
Counted loops with range(), conditional loops, accumulator pattern, sentinel-controlled loops, off-by-one errors
# Sum of numbers 1..n using accumulator pattern def sum_to(n): total = 0 for i in range(1, n + 1): total += i return total print(sum_to(100)) # 5050 (Gauss)
📖 Lesson Notes — Ontario Curriculum

Iteration — repeating a block of code — is what makes a computer dramatically more powerful than a calculator. A for loop iterates over a known sequence (a list, a range, a string, a file), and the loop variable takes on each element in turn. A while loop continues as long as a Boolean condition holds, and is used when the number of iterations is not known in advance (e.g., reading until end-of-file, or polling for user input). Choosing between them is a design decision: for when you can describe what to iterate over; while when you need to describe when to stop.

The accumulator pattern initialises a result variable outside the loop, updates it inside the loop, and returns it after the loop. The accumulator can be a number (sum, product, count), a Boolean (any/all), a list (filter, map), or a dictionary (frequency count). Recognising this pattern lets students translate plain-English questions ("how many students passed?") into code mechanically.

The sentinel-controlled loop is a while idiom where a special "stop" value (like "quit", -1, or empty string) ends the loop. The sentinel must be a value that could not appear as legitimate data; choosing it carelessly is a famous source of bugs.

  • Loop body — the indented block that repeats.
  • Loop variable / index — the variable that changes each iteration.
  • Accumulator — a variable that builds up a result across iterations.
  • Sentinel — a special input value that signals the end of input.
  • Off-by-one error — looping one too few or one too many times, often from range(n) vs range(n+1).
  • Infinite loop — a while whose condition never becomes false.

Worked example — tracing a nested loop

def count_pairs_summing_to(target, nums):
    count = 0
    for i in range(len(nums)):
        for j in range(i + 1, len(nums)):  # j > i avoids dupes
            if nums[i] + nums[j] == target:
                count += 1
    return count

print(count_pairs_summing_to(10, [1, 9, 4, 6, 5, 5]))  # 3

Pairs that sum to 10: (1,9), (4,6), (5,5). The outer loop runs n times; the inner loop runs n-i-1 times for iteration i. Total comparisons = n(n-1)/2, so the time complexity is O(n²). Memory is O(1) — only a counter is stored. A faster O(n) algorithm uses a hash set to remember target - nums[i] values; this trade-off (time vs space) appears repeatedly in Unit 4.

Worked example — desk-checking a while loop

n = 12
steps = 0
while n > 1:
    if n % 2 == 0:
        n = n // 2
    else:
        n = 3*n + 1
    steps += 1
print(steps)  # 9 — the Collatz sequence

Trace: 12 → 6 → 3 → 10 → 5 → 16 → 8 → 4 → 2 → 1. Nine iterations. Whether this loop terminates for every positive n is the famous unsolved Collatz conjecture — a reminder that loop termination is not always trivial to prove.

  • Off-by-one: range(1, n) goes 1 to n-1, not 1 to n. Use range(1, n+1) to include n.
  • Modifying a list while iterating over it — leads to skipped or duplicated elements.
  • Infinite loops because the loop variable is never updated inside while.
  • Initialising an accumulator inside the loop instead of outside — resets it every iteration.
  • Using break in deeply nested loops without a clear exit strategy.

1.5 — Program Tracing and Desk Checks
Variable trace tables, dry runs, predicting output, the IPO (Input-Process-Output) chart
📖 Lesson Notes — Ontario Curriculum

Tracing — manually simulating the execution of a program with pen and paper — is one of the most important diagnostic skills a programmer develops. A trace table has one column per variable plus an "output" column, and one row per statement executed. By filling it in step-by-step, the student verifies that what the code actually does matches what they think it does. The technique is invaluable on tests, in code reviews, and during debugging, because it forces explicit reasoning about every state change.

The IPO chart (Input-Process-Output) is a higher-level planning tool used before coding. It lists what data enters the program, what transformations are applied, and what data leaves. IPO charts feed into structure charts, which decompose the "Process" box into a tree of sub-functions. Together they constitute the documentation a maintainer needs to understand a program's purpose without reading every line.

Tracing also reveals the difference between three classes of error: a syntax error prevents the program from running; a runtime error aborts the program mid-execution (e.g., divide-by-zero, index out of range); and a logic error lets the program complete but produces the wrong answer. Logic errors are the hardest to find precisely because the program "works" — tracing exposes them by comparing predicted output against actual output.

  • Trace table — a table tracking variable values line-by-line through execution.
  • Dry run — executing a program on paper without a computer.
  • IPO chart — a 3-column diagram listing Input, Process, Output.
  • Structure chart — a hierarchical diagram showing how modules call one another.
  • Syntax error — code that violates language grammar; caught by the parser.
  • Runtime error — an exception raised during execution.
  • Logic error — the program runs but produces incorrect output.
  • Invariant — a condition that holds true at a specific point in every iteration.

Worked example — trace this loop

def mystery(n):
    a, b = 0, 1
    while a < n:
        a, b = b, a + b
    return a

print(mystery(20))

Trace table:

iter | a  | b  | a < 20?
-----+----+----+--------
 0   | 0  | 1  | True
 1   | 1  | 1  | True
 2   | 1  | 2  | True
 3   | 2  | 3  | True
 4   | 3  | 5  | True
 5   | 5  | 8  | True
 6   | 8  | 13 | True
 7   | 13 | 21 | True
 8   | 21 | 34 | False → exit
return 21

The function returns the smallest Fibonacci number ≥ n. The loop invariant is "a and b are consecutive Fibonacci numbers". Time complexity is O(log φ · n) ≈ O(log n) because Fibonacci numbers grow exponentially. Memory is O(1).

  • Updating two variables sequentially when simultaneous update is required: a = b; b = a + b is not the same as a, b = b, a + b.
  • Forgetting to record output rows in the trace table — leads to missed off-by-one errors.
  • Tracing only a couple of iterations and assuming the pattern continues.
  • Confusing pre-condition (checked before the loop body) with post-condition (after).
  • Building an IPO chart with "process" steps that are vague verbs like "do the math" — every step should be testable.

2.1 — 1-D Lists (Arrays): Indexing, Slicing, Mutation
List literal syntax, zero-based indexing, negative indices, slice notation, common methods (.append, .pop, .insert, .remove)
scores = [82, 76, 91, 68, 88] print(scores[0]) # 82 (first) print(scores[-1]) # 88 (last) print(scores[1:4]) # [76, 91, 68] scores.append(95) # mutate in place avg = sum(scores) / len(scores)
📖 Lesson Notes — Ontario Curriculum

A list (in many languages called an array) is the single most useful data structure in computing. It stores an ordered, zero-indexed collection of elements in contiguous (or logically contiguous) memory, supporting O(1) random access via an integer index. Python lists are mutable and heterogeneous (they may hold mixed types), unlike Java arrays which are fixed-size and homogeneous. Python's list is implemented as a dynamic array of object references — appending amortises to O(1), but inserting at the front (insert(0, x)) is O(n) because every element must shift right.

Indexing in Python starts at 0 and runs to len(lst) - 1; negative indices count from the end (lst[-1] is the last element). Slicing with lst[start:stop:step] returns a new list — a critical fact, because modifying the slice does not affect the original. The half-open interval convention (start inclusive, stop exclusive) means lst[1:4] contains exactly 3 elements, not 4. This eliminates many off-by-one bugs once you internalise it: the length of a slice is always stop - start.

Mutability has consequences. Assigning b = a creates an alias, not a copy — both names refer to the same underlying list. b.append(7) changes what a sees. To make a shallow copy, use a[:], list(a), or a.copy(). For nested lists, use copy.deepcopy.

  • Index — a zero-based integer position into a list.
  • Slicing — extracting a sub-range with lst[start:stop:step].
  • Mutable — can be changed in place after creation (lists, dicts, sets).
  • Immutable — cannot be changed in place (strings, tuples, ints, floats).
  • Aliasing — two names referring to the same object.
  • Shallow copy — copies the top-level list but shares nested objects.
  • Amortised complexity — average cost per operation over a sequence of operations.

Worked example — building a histogram

def histogram(scores, bin_size=10):
    bins = [0] * (100 // bin_size + 1)   # 11 bins for 0-100
    for s in scores:
        bins[s // bin_size] += 1
    return bins

print(histogram([55, 62, 68, 71, 73, 84, 91, 91, 99]))
# [0, 0, 0, 0, 0, 1, 2, 2, 1, 2, 1]

Constructing the bins list is O(k) where k is the number of bins; the loop runs through n scores doing O(1) work each, so the total complexity is O(n + k). This is asymptotically optimal for counting because every score must be examined at least once. The space cost is O(k).

  • Index out of range: accessing lst[len(lst)] raises IndexError. Last valid index is len(lst) - 1.
  • Aliasing bug: b = a then thinking b is a copy. Use b = a[:] or b = list(a).
  • Multiplying a list of lists: [[0]*3]*3 creates three references to the same inner list. Use a list comprehension instead.
  • Calling lst.append expecting a new list back — it returns None.
  • Repeatedly inserting at the front when the desired order can be built by appending and reversing once at the end.

2.2 — 2-D Lists / Multi-Dimensional Arrays
Lists of lists, row/column indexing, nested loops to traverse a grid, building a tic-tac-toe or seating chart
# 3 x 3 game board (None = empty) board = [ [None, "X", None], [None, "O", None], ["X", None, None] ] for row in board: for cell in row: print(cell if cell else ".", end=" ") print()
📖 Lesson Notes — Ontario Curriculum

Two-dimensional data — grids, matrices, images, game boards, spreadsheets — show up everywhere in computing. In Python the natural representation is a list of lists: the outer list holds rows, each row is itself a list of cells. To address cell (r, c) we write grid[r][c]: the first index selects the row, the second selects the column. This convention (row-major) matches mathematical matrix notation and how C/Java store 2-D arrays in memory.

Traversing a 2-D structure almost always uses nested loops: the outer loop walks rows, the inner loop walks columns. The order matters for performance in compiled languages because of CPU cache lines, but at the Python level it changes only the order in which cells are visited. The total work for a full traversal of an r × c grid is O(r·c), which is linear in the number of cells.

A frequent pitfall is creating an "empty" 2-D grid by multiplying a row: [[0]*c]*r looks reasonable but produces a list where every row is the same object. Writing to grid[0][1] also writes to grid[1][1], grid[2][1], and so on. The correct idiom is a nested list comprehension: [[0]*c for _ in range(r)], which evaluates the inner expression independently for each row.

  • 2-D list / matrix — a list whose elements are themselves lists of equal length.
  • Row-major order — storing elements row-by-row in memory; the natural Python layout.
  • Nested loop — a loop inside another loop, giving multiplicative iteration counts.
  • Dimensions — the number of rows and columns, often written r × c.
  • Sparse vs dense — whether most cells hold meaningful data (dense) or default values (sparse).

Worked example — Conway's Life one-step update

def neighbour_count(grid, r, c):
    rows, cols = len(grid), len(grid[0])
    count = 0
    for dr in (-1, 0, 1):
        for dc in (-1, 0, 1):
            if dr == 0 and dc == 0:
                continue
            nr, nc = r + dr, c + dc
            if 0 <= nr < rows and 0 <= nc < cols:
                count += grid[nr][nc]
    return count

def step(grid):
    rows, cols = len(grid), len(grid[0])
    new = [[0]*cols for _ in range(rows)]   # independent rows!
    for r in range(rows):
        for c in range(cols):
            n = neighbour_count(grid, r, c)
            if grid[r][c] == 1 and n in (2, 3):
                new[r][c] = 1
            elif grid[r][c] == 0 and n == 3:
                new[r][c] = 1
    return new

For each of the r·c cells we examine 8 neighbours, all O(1) work, so a single step costs O(r·c). Bounds checking (0 <= nr < rows) handles edge cells without special cases. The new grid is built fresh to avoid reading half-updated state.

  • Initialising with [[0]*c]*r — every row aliases the same list. Use a comprehension.
  • Swapping r and c in the index: grid[c][r] when you meant grid[r][c].
  • Updating the grid in place while reading it — neighbouring cells see the new state, not the old.
  • Forgetting to bounds-check at edges, producing IndexError or wraparound from negative indices.
  • Assuming all rows have the same length — they don't, unless you enforce it.

2.3 — Dictionaries (Maps) and Records
Key-value pairs, lookup vs. linear search, dictionaries as records, nested dictionaries
student = { "name": "Hopper", "id": 100451, "grades": {"ICS4U": 94, "MCV4U": 88} } print(student["grades"]["ICS4U"]) # 94
📖 Lesson Notes — Ontario Curriculum

A dictionary (also called a map, hash table, or associative array) maps keys to values. Where a list is indexed by integer position, a dictionary is indexed by arbitrary immutable values — most commonly strings. Internally, Python uses an open-addressed hash table, which gives expected O(1) lookup, insertion, and deletion. Compare this with a linear search through a list of n records (O(n)): on a roster of 10,000 students, a dictionary lookup is roughly 10,000× faster.

Keys must be hashable, which in practice means immutable: strings, numbers, and tuples-of-immutables work; lists and dicts do not. The hash function maps each key to a bucket; collisions are resolved by probing. When the table fills past a load factor (~⅔), it is resized — an amortised cost that does not affect the O(1) expected bound.

Dictionaries shine as records: bundling related fields under named keys (student["name"], student["gpa"]) makes code self-documenting compared with parallel lists or magic tuple positions. They are also the workhorse for counting (frequency tables), grouping (bucketing by key), memoisation (caching computed results), and graph adjacency (mapping vertex → list of neighbours).

  • Key-value pair — the unit of storage in a dict.
  • Hash function — maps a key to an integer bucket index.
  • Hashable — has a stable hash; in Python, immutable types are hashable.
  • Collision — two keys hash to the same bucket.
  • Load factor — ratio of stored entries to bucket count.
  • Memoisation — caching the result of an expensive function in a dict.

Worked example — word-frequency counter

def word_frequencies(text):
    freq = {}
    for word in text.lower().split():
        word = word.strip(".,!?;:\"'")
        if word:
            freq[word] = freq.get(word, 0) + 1
    return freq

text = "To be, or not to be: that is the question."
print(word_frequencies(text))
# {'to': 2, 'be': 2, 'or': 1, 'not': 1, 'that': 1, 'is': 1, 'the': 1, 'question': 1}

The .get(word, 0) idiom returns 0 when the key is missing, avoiding a KeyError. The total cost is O(n) where n is the number of words: each iteration does O(1) hash work, and the final dict has at most n entries. For comparison, doing the same job with a list of (word, count) tuples and a linear search would be O(n²) — quadratically slower as the text grows.

Worked example — memoised Fibonacci

cache = {}
def fib(n):
    if n in cache:
        return cache[n]
    if n < 2:
        return n
    cache[n] = fib(n-1) + fib(n-2)
    return cache[n]

print(fib(50))  # 12586269025 — instant with memoisation

Without the cache, naive recursive Fibonacci is O(φⁿ) ≈ O(1.618ⁿ); with memoisation each n is computed once and stored, dropping the cost to O(n) time and O(n) space. This is the same principle that powers dynamic programming.

  • Using a list as a dict key — raises TypeError: unhashable type. Convert to tuple first.
  • Accessing a missing key with d[k] — raises KeyError. Use d.get(k, default) or if k in d.
  • Iterating over d assuming it yields values — it yields keys. Use d.values() or d.items().
  • Mutating a dict while iterating over it — raises RuntimeError. Iterate over list(d.keys()) instead.
  • Assuming order: insertion order is preserved since Python 3.7, but never rely on sorted order.

2.4 — Strings as Sequences
Strings are immutable sequences; common methods (.split, .join, .strip, .lower, .replace, .find); slicing and reversal
📖 Lesson Notes — Ontario Curriculum

A string is an immutable sequence of characters. Many of the operations that work on lists — indexing, slicing, iteration, len, in — also work on strings. The crucial difference is mutability: you cannot write s[0] = 'X' because strings cannot be modified in place. Every method that "changes" a string actually returns a new string. For long-running text processing this matters: building a result by repeated concatenation (result = result + char) is O(n²) because each + copies the entire accumulated string. The idiomatic fix is to collect parts in a list and call "".join(parts) at the end, which is O(n).

Python strings are Unicode (UTF-8 internally on most platforms); each character is a code point, not a byte. This means len("café") is 4, not 5 — but writing it to a file in UTF-8 produces 5 bytes because é takes two bytes. Awareness of the text/bytes distinction prevents the classic "works on my machine until the input contains an accent" bug.

String methods form a small standard library every Grade 12 student should memorise: .split(sep) tokenises into a list, sep.join(lst) is its inverse, .strip() removes leading/trailing whitespace, .lower() and .upper() change case, .replace(old, new) substitutes, .find(sub) returns the first index or -1, .startswith/.endswith are useful for parsing.

  • Immutable — value cannot be modified after creation.
  • Code point — a Unicode integer identifying one character.
  • Encoding — the byte sequence used to store a code point (UTF-8, UTF-16, ASCII).
  • Substring — a contiguous slice of a string.
  • Tokenisation — splitting a string into meaningful units (words, fields).
  • Palindrome — a string that reads the same forwards and backwards.

Worked example — palindrome check and CSV parse

def is_palindrome(s):
    s = ''.join(c.lower() for c in s if c.isalnum())
    return s == s[::-1]   # [::-1] reverses

print(is_palindrome("A man, a plan, a canal: Panama"))  # True
print(is_palindrome("ICS4U"))                            # False

def parse_csv_row(row):
    return [field.strip() for field in row.split(',')]

print(parse_csv_row(" Ada , 17, 3.85 "))  # ['Ada', '17', '3.85']

The palindrome routine first filters out non-alphanumeric characters and lowercases each one — O(n). The slice s[::-1] creates a reversed copy in O(n), and the equality check is O(n). Total: O(n) time, O(n) space. The CSV parser uses .split(',') followed by .strip() on each field; this is also O(n) but ignores the subtleties of quoted fields — the csv module handles those properly.

  • Trying to assign to an index: s[0] = 'X' raises TypeError.
  • Building a result with += in a loop — O(n²). Use "".join(list_of_parts).
  • Confusing .find (returns -1 on miss) with .index (raises ValueError).
  • Treating str.split() with no argument as the same as .split(' ') — they differ on runs of whitespace.
  • Forgetting that "7" < "10" is True (lexicographic), not False (numeric).

2.5 — File I/O: Reading and Writing Text Files
open(), context managers (with statement), reading lines, writing CSV-like data
# Read a roster file, output the average mark with open("roster.csv") as f: total, count = 0, 0 for line in f: name, mark = line.strip().split(",") total += int(mark) count += 1 print(f"Class average: {total/count:.1f}")
📖 Lesson Notes — Ontario Curriculum

Programs need persistence — the ability to save data between runs. Files are the simplest form of persistence: a named, byte-addressed region on disk. Python's open(path, mode) returns a file object; mode is "r" for read, "w" for write (truncates existing content), "a" for append, and "b" appended for binary. A file object is iterable in line-by-line fashion, which means for line in f reads one line at a time without loading the entire file into memory — essential for large data sets.

The with statement is a context manager that guarantees the file is closed even if an exception is raised inside the block. Forgetting to close a file can leak operating-system handles and, on Windows, prevent other programs from accessing it. with is the modern, idiomatic way to handle files; bare open without close is a code-review red flag.

Text files store lines terminated by \n (Unix) or \r\n (Windows); Python's text-mode I/O normalises these to \n on read and the platform default on write. The CSV (comma-separated values) format is the simplest tabular interchange format: one row per line, fields separated by commas. The csv module in the standard library handles quoted fields, escaped commas, and varied dialects — students should prefer it over hand-rolled .split(',') once data gets messy.

  • File handle — the OS-level reference returned by open.
  • Mode — "r", "w", "a", optionally with "b" for binary or "+" for read/write.
  • Context manager — an object usable with with that defines setup/teardown.
  • Buffering — accumulating writes before flushing to disk for efficiency.
  • EOF (end-of-file) — signalled by an empty string from read or by the loop ending.
  • CSV — a plain-text tabular format.

Worked example — read, transform, write

import csv

def upgrade_marks(in_path, out_path, bonus=5):
    with open(in_path, newline='') as fin, \
         open(out_path, 'w', newline='') as fout:
        reader = csv.DictReader(fin)
        writer = csv.DictWriter(fout, fieldnames=reader.fieldnames)
        writer.writeheader()
        for row in reader:
            row['mark'] = min(100, int(row['mark']) + bonus)
            writer.writerow(row)

upgrade_marks('marks.csv', 'marks_curved.csv')

Two files are opened in a single with; both are guaranteed to close. The reader and writer stream line-by-line so memory use is O(1) regardless of file size. Total time is O(n) in the number of rows. The min(100, …) guard caps marks so a bonus never pushes a 98 to 103. Real Ontario reporting policies vary; this is just an illustration.

  • Opening a file in "w" mode "to check it exists" — it will be truncated to empty.
  • Forgetting with and not calling .close(), leaking handles.
  • Reading the whole file into memory with f.read() when streaming would suffice.
  • Forgetting that line from iteration includes the trailing \n; use line.rstrip("\n").
  • Hand-parsing CSV with split(',') when fields may contain commas inside quotes.
  • Hard-coding absolute paths that break on another machine — use relative paths or pathlib.

2.6 — Classes and Objects (Introduction to OOP)
Defining a class, attributes, methods, the __init__ constructor, encapsulation; objects vs. records
class Student: def __init__(self, name, grade): self.name = name self.grade = grade def promote(self): self.grade += 1 def __str__(self): return f"{self.name} (Grade {self.grade})" s = Student("Turing", 11) s.promote() print(s) # Turing (Grade 12)
📖 Lesson Notes — Ontario Curriculum

Object-oriented programming (OOP) bundles related data and the operations on that data into a single unit called a class. A class is a blueprint; an instance (or object) is a concrete realisation in memory. The instance's attributes hold its data, and its methods are functions that operate on that data. The __init__ method (the constructor) runs automatically when a new instance is created, initialising attributes from the arguments passed.

OOP's main benefits are encapsulation (data and behaviour live together, so external code does not need to know representation details), abstraction (a clean interface hides complexity), and (once we add inheritance) reuse. A Student object knows its own name and grade and knows how to promote itself — calling code just writes s.promote() rather than manipulating fields directly. This makes the program easier to read, easier to change (the implementation can swap out without touching callers), and easier to test (each class can be unit-tested in isolation).

The self parameter is the explicit reference to the current instance. In Python you must list it; Java's this is implicit. The __str__ "dunder" method (double-underscore on both sides) defines how the object is converted to a string for print or str(). Providing a meaningful __str__ is a small effort that pays huge dividends during debugging.

  • Class — a blueprint defining attributes and methods.
  • Instance / object — a concrete value created from a class.
  • Attribute — a piece of data stored on an instance (or class).
  • Method — a function defined inside a class.
  • Constructor — the __init__ method that runs at creation.
  • Encapsulation — bundling state with the methods that change it.
  • Dunder method — a special method named with double underscores (e.g. __str__, __eq__).

Worked example — a BankAccount class

class BankAccount:
    def __init__(self, owner, opening_balance=0):
        self.owner = owner
        self._balance = opening_balance        # leading _ = "private"

    def deposit(self, amount):
        if amount <= 0:
            raise ValueError("amount must be positive")
        self._balance += amount

    def withdraw(self, amount):
        if amount > self._balance:
            raise ValueError("insufficient funds")
        self._balance -= amount

    def balance(self):
        return self._balance

    def __str__(self):
        return f"{self.owner}: ${self._balance:,.2f}"

a = BankAccount("Ada", 100)
a.deposit(50)
a.withdraw(30)
print(a)            # Ada: $120.00
print(a.balance())  # 120

Each operation runs in O(1) and uses O(1) extra memory per account. The leading underscore on _balance is a convention signalling "do not access from outside the class" — Python does not enforce it but reviewers do. Encapsulation here prevents callers from corrupting the balance directly; the only legal mutations go through deposit and withdraw, which validate their arguments.

  • Forgetting self as the first parameter of every instance method — produces confusing TypeErrors.
  • Defining attributes outside __init__ (as class variables) and then mutating them — they are shared across all instances.
  • Using a mutable default argument: def __init__(self, items=[]) — every instance shares the same list!
  • Returning self from a setter and then assigning obj = obj.set_x(5) — works, but confuses students about mutation vs assignment.
  • Comparing objects with == without overriding __eq__ — falls back to identity comparison.

3.1 — Defining and Calling Functions
Function definition, parameters vs. arguments, return values, docstrings, the DRY principle
def celsius_to_fahrenheit(c): """Convert °C to °F.""" return c * 9/5 + 32 def describe_temp(c): f = celsius_to_fahrenheit(c) return f"{c}°C = {f:.1f}°F"
📖 Lesson Notes — Ontario Curriculum

A function is a named, reusable unit of computation. It takes parameters (the names listed in the definition), receives arguments (the values supplied at the call site), and may produce a return value. The discipline of factoring code into small, well-named functions is the single largest contributor to program quality. Functions enable the DRY principle (Don't Repeat Yourself), allow unit testing in isolation, and make the overall logic readable as a sequence of intentions rather than a wall of statements.

Every function should have one clear job, named with a verb phrase. celsius_to_fahrenheit is good; doStuff is not. A function that returns a value (a pure function) is easier to reason about than one that prints or mutates external state, because its behaviour is determined entirely by its arguments. Whenever possible, prefer returning a value to printing one — the caller can decide what to do with it.

Docstrings (triple-quoted strings immediately after the def line) are documentation that lives with the code. They are accessible at runtime via help(func) and are read by IDEs to provide tooltips. PEP 257 recommends one-line docstrings for trivial functions and structured "Args/Returns/Raises" sections for complex ones.

  • Function definition — the def name(params): header and body.
  • Parameter — a name in the function header.
  • Argument — a value passed at a call site.
  • Return value — the value produced by the return statement.
  • Pure function — output depends only on inputs; no side effects.
  • Side effect — modification of external state (I/O, globals, mutating arguments).
  • DRY — Don't Repeat Yourself; share logic via functions.

Worked example — refactoring to functions

# BEFORE — repetitive, hard to test
m1, m2, m3 = 88, 76, 91
avg1 = (m1 + m2 + m3) / 3
print(f"Avg 1: {avg1:.1f}")
m4, m5, m6 = 65, 72, 68
avg2 = (m4 + m5 + m6) / 3
print(f"Avg 2: {avg2:.1f}")

# AFTER — one function, one responsibility
def average(marks):
    """Return the arithmetic mean of a non-empty list of numbers."""
    if not marks:
        raise ValueError("cannot average an empty list")
    return sum(marks) / len(marks)

def report(label, marks):
    print(f"{label}: {average(marks):.1f}")

report("Avg 1", [88, 76, 91])  # Avg 1: 85.0
report("Avg 2", [65, 72, 68])  # Avg 2: 68.3

average is now O(n) in the length of the list, pure (no I/O), and testable in isolation: a unit test can assert average([1,2,3]) == 2.0 without capturing print output. report handles the side effect (printing) separately. Separating concerns this way is fundamental to modular design.

  • Forgetting return — Python silently returns None, often producing puzzling downstream errors.
  • Defining a function inside a loop unnecessarily, paying the definition cost on every iteration.
  • Naming a function the same as a built-in (sum, list, id) — shadows the built-in.
  • Mixing concerns: a function that computes and prints and writes to a file is hard to test.
  • Using global variables inside a function instead of accepting them as parameters.

3.2 — Parameters: Positional, Keyword, Default, and *args
Default arguments, named arguments, variable-length argument lists, pass-by-object-reference behavior
📖 Lesson Notes — Ontario Curriculum

Python's parameter system is more flexible than the simple positional parameters found in many languages. Positional arguments are matched to parameters by their order in the call. Keyword arguments are matched by name and may appear in any order: connect(host="hub.school.ca", port=22) reads better than connect("hub.school.ca", 22). Default values let parameters be omitted: def greet(name, greeting="Hello") may be called as greet("Ada"). Variable-length parameters (*args, **kwargs) collect any extra positional or keyword arguments into a tuple or dict.

Python uses pass-by-object-reference (sometimes called "pass-by-assignment"): the function parameter is bound to the same object the caller passed. For immutable types (int, float, str, tuple) this is observationally identical to pass-by-value — the function cannot affect the caller's variable. For mutable types (list, dict, set, custom objects) the function can mutate the object in place, and the caller sees the change. Reassigning the parameter (x = something_new) does not affect the caller because reassignment only rebinds the local name.

This distinction is the leading source of confusion for students who learned "pass-by-value vs pass-by-reference" in another course. The Python rule is simpler once internalised: names are bindings to objects; arguments share the same binding. Method calls and indexed assignment go through the object and are visible; rebinding the local name is not.

  • Positional parameter — matched to argument by position.
  • Keyword parameter — matched by name.
  • Default value — used when no argument is supplied.
  • *args / **kwargs — collect surplus arguments.
  • Pass-by-object-reference — the parameter and argument refer to the same object.
  • Side effect — visible change to a mutable argument or external state.

Worked example — the mutable-default-argument trap

# BUG — the default list is shared across calls
def append_bad(item, lst=[]):
    lst.append(item)
    return lst

print(append_bad(1))   # [1]
print(append_bad(2))   # [1, 2]  ← surprise!
print(append_bad(3))   # [1, 2, 3]

# FIX — sentinel default, create a fresh list inside
def append_ok(item, lst=None):
    if lst is None:
        lst = []
    lst.append(item)
    return lst

print(append_ok(1))    # [1]
print(append_ok(2))    # [1]

The mutable default [] is evaluated once at function-definition time, not at every call. Every invocation that omits the argument therefore shares the same list. The None sentinel pattern is the canonical Python fix; this idiom appears constantly in production code.

Worked example — pass-by-object-reference

def add_one(lst):
    lst.append(1)     # mutation — caller sees it
    lst = [99]        # rebinding — caller does NOT see it

data = [10, 20]
add_one(data)
print(data)           # [10, 20, 1]

The append modifies the object both names share. The reassignment changes only the local lst; the caller's data still points to the original (now mutated) list.

  • Mutable default arguments — single most famous Python trap.
  • Putting positional parameters after defaulted ones — SyntaxError.
  • Expecting reassignment of a parameter to affect the caller — it doesn't.
  • Calling with positional args in the wrong order when keyword args would be clearer.
  • Overusing **kwargs — opaque to readers and to the type checker.

3.3 — Variable Scope and the LEGB Rule
Local, Enclosing, Global, Built-in scope; the global keyword; why mutable globals are dangerous
x = 10 # global def outer(): x = 5 # enclosing (relative to inner) def inner(): x = 1 # local print(x) # 1 inner() print(x) # 5 outer() print(x) # 10
📖 Lesson Notes — Ontario Curriculum

The scope of a name is the region of source code in which the name is visible. Python uses the LEGB rule to resolve names: it searches Local (current function), then Enclosing (any surrounding functions), then Global (module-level), then Built-in (the always-available names like print and len). The first match wins, and the search stops there. Understanding LEGB explains both why a name resolves the way it does and why some name-related errors appear.

Assigning to a name inside a function makes that name local to the function, even if a global with the same name exists. The global keyword overrides this, telling Python "this name refers to the module-level variable, not a new local one". The nonlocal keyword does the analogous job for enclosing scopes (used in closures). Heavy use of global is considered a smell because it makes functions dependent on hidden state — pass values in as parameters and return results explicitly.

Scope rules also explain the UnboundLocalError: if any line in a function assigns to x, then every reference to x in that function is treated as local, even references that physically precede the assignment. Reading x before the assignment runs raises UnboundLocalError rather than falling back to the global.

  • Scope — the region of code in which a name is visible.
  • Local scope — names defined inside the current function.
  • Enclosing scope — names in outer functions when nested.
  • Global scope — names at module level.
  • Built-in scope — names provided by Python itself.
  • LEGB rule — order of scope resolution.
  • Shadowing — a local name hiding an outer name with the same identifier.

Worked example — the UnboundLocalError trap

counter = 0          # global

def bump_bad():
    counter += 1     # UnboundLocalError!
    print(counter)

def bump_global():
    global counter
    counter += 1
    print(counter)

def bump_clean(c):   # preferred: pass it in
    return c + 1

bump_global()        # 1
bump_global()        # 2
counter = bump_clean(counter)
print(counter)       # 3

bump_bad contains an assignment to counter, which marks the name as local for the entire function — so the read on the right-hand side fails before the write happens. bump_global opts in explicitly. bump_clean is the preferred design: parameters in, return values out, no hidden coupling. Each pattern is O(1).

  • Assigning to a global name without declaring global — creates a local accidentally.
  • Using global for everything — defeats the purpose of functions.
  • Reusing a built-in name (list, str, id, type) as a variable, then being unable to use the built-in.
  • Expecting a loop variable to remain accessible only inside the loop — in Python it persists after the loop ends.
  • Confusing nonlocal (enclosing function scope) with global (module scope).

3.4 — Recursion: Base Cases and Recursive Cases
Designing the base case, the recursive call, the call stack, and the equivalence with iteration
def factorial(n): if n <= 1: # base case return 1 return n * factorial(n-1) # recursive case def fib(n): if n < 2: return n return fib(n-1) + fib(n-2)
📖 Lesson Notes — Ontario Curriculum

A recursive function is one that calls itself on a smaller version of the same problem, until it reaches a base case that can be answered directly. Every recursive solution has two essential parts: the base case (which must terminate the recursion) and the recursive case (which makes progress toward the base case). Forgetting either one is catastrophic — missing the base case produces infinite recursion until the interpreter raises RecursionError: maximum recursion depth exceeded; missing the progress step does the same.

Recursion is implemented via the call stack: each function call pushes a new stack frame holding the local variables and the return address; each return pops the top frame. A recursive call of depth d uses O(d) stack memory. Python's default recursion limit is 1000, which means deeply recursive solutions on big inputs can hit the wall even when the algorithm is correct. For problems that lend themselves to iteration (like factorial), the iterative form uses O(1) stack.

Recursion shines on problems that are themselves recursively structured: tree traversal, divide-and-conquer sorting, fractals, parsing nested expressions, the Towers of Hanoi, the Euclidean GCD algorithm. For "naturally linear" problems it is often a less efficient way to express what a loop does. Knowing when to choose recursion is a Grade 12 judgement call.

  • Base case — the smallest case answered without further recursion.
  • Recursive case — the call(s) on a smaller sub-problem.
  • Call stack — the LIFO data structure holding active function calls.
  • Stack frame — one entry on the call stack.
  • Direct recursion — a function calls itself.
  • Indirect recursion — function A calls B which calls A.
  • Tail recursion — the recursive call is the very last action; some languages optimise it to a loop (Python does not).

Worked example — Euclidean GCD and exponentiation

def gcd(a, b):
    if b == 0:               # base case
        return a
    return gcd(b, a % b)     # recursive case

print(gcd(252, 105))         # 21

def power(base, exp):
    if exp == 0:
        return 1
    if exp % 2 == 0:
        half = power(base, exp // 2)
        return half * half
    return base * power(base, exp - 1)

print(power(2, 10))          # 1024

gcd reduces the problem dramatically each call — by Lamé's theorem the depth is O(log(min(a, b))), so the algorithm runs in O(log n) time and O(log n) stack space. power uses fast exponentiation: by squaring on even exponents and decrementing on odd ones, it makes O(log exp) calls instead of the O(exp) of naive repeated multiplication. Both are textbook examples of how recursion captures the algorithm's structure cleanly.

Worked example — the naive Fibonacci performance trap

def fib(n):
    if n < 2: return n
    return fib(n-1) + fib(n-2)

# fib(40) takes ~1 second
# fib(50) takes ~2 minutes
# Why? T(n) = T(n-1) + T(n-2) + O(1)
# This grows as O(φⁿ) ≈ O(1.618ⁿ)

The naive recurrence makes exponentially many calls because fib(n-2) is computed inside both branches. Memoisation (storing already-computed results in a dict) drops it to O(n). This is one of the cleanest demonstrations of why algorithm choice matters even when both versions are "correct".

  • Forgetting the base case — infinite recursion, RecursionError.
  • Recursive case that does not make progress toward the base (e.g., calling f(n) instead of f(n-1)).
  • Using recursion where iteration is simpler and uses O(1) stack.
  • Naively recursing over overlapping sub-problems without memoisation — exponential blow-up.
  • Returning the wrong value from the base case (e.g., returning 0 for empty-product when 1 is correct).

3.5 — Top-Down Design and Stepwise Refinement
Decompose a problem into modules; structure charts; bottom-up integration; library reuse
📖 Lesson Notes — Ontario Curriculum

Top-down design is a problem-solving strategy: start with the highest-level statement of what the program must do, and recursively decompose it into sub-tasks until each sub-task is small enough to code in a few lines. The output of this process is a structure chart, a tree diagram where each node represents one function and arrows show "calls". Top-down design tames complexity because the programmer reasons about one level at a time; when a sub-task is implemented, the level above can use it without re-examining its internals.

The companion technique is stepwise refinement: take a pseudocode line like "process all students" and refine it into "for each student, validate the record, compute the GPA, write to the report". Each refinement adds detail without changing the structure above. The final code mirrors the chart — small, named functions that compose into the whole.

A complementary strategy, bottom-up design, builds reusable utilities first and assembles them into a larger system. In practice, professional software engineers oscillate between the two: top-down to clarify what is needed, bottom-up to leverage what already exists in standard libraries or previous projects. Library reuse — using the standard library, well-maintained third-party packages, or in-house modules — is the most cost-effective form of bottom-up design, because someone has already paid the cost of writing and debugging the code.

  • Top-down design — decomposing a problem into sub-problems.
  • Bottom-up design — building reusable components first, then composing them.
  • Stepwise refinement — iteratively replacing a vague step with a detailed sub-procedure.
  • Structure chart — tree diagram of functions and their callers.
  • Module / library — a file (or package) of related functions and classes.
  • Cohesion — how focused a module is on a single purpose (high is good).
  • Coupling — how dependent modules are on each other (low is good).

# Top-level intention

    students = load_students(input_csv)
    stats    = compute_class_stats(students)
    write_pdf(output_pdf, students, stats)

The complexity of each sub-task can be analysed separately — load_students is O(n), compute_class_stats is O(n) (or O(n log n) if it sorts), and write_pdf is O(n) for the body of the document.

  • Writing one giant main function with everything inline — impossible to test or reuse.
  • Skipping the design step and "just coding" — leads to refactors that should have been avoided.
  • Excessive layers (a function for every two lines) — over-engineered and harder to follow.
  • High coupling: one module reaches into another's internals instead of using its public API.
  • Reinventing standard-library functionality (don't write your own sort or csv parser unless that's the assignment).

4.1 — Linear Search and Binary Search
Sequential lookup vs. divide-and-conquer; precondition: binary search requires a sorted list; off-by-one issues with bounds
def binary_search(arr, target): lo, hi = 0, len(arr) - 1 while lo <= hi: mid = (lo + hi) // 2 if arr[mid] == target: return mid elif arr[mid] < target: lo = mid + 1 else: hi = mid - 1 return -1 # not found
📖 Lesson Notes — Ontario Curriculum

Searching — finding whether and where a target value lives in a collection — is one of the most-performed operations in computing. The two textbook algorithms illustrate a fundamental complexity gap. Linear search examines elements one at a time from the start; it works on any sequence and requires no preconditions. Its worst case is O(n) comparisons. Binary search exploits a sorted array by repeatedly halving the search interval; its worst case is O(log n) comparisons. The difference is dramatic: searching a million-element list takes up to one million comparisons linearly, but only twenty binarily.

Binary search carries a strict precondition: the input must be sorted in the comparison's order. Running binary search on an unsorted list returns nonsense. If the data will be searched many times, it is worth sorting once (O(n log n)) and then searching many times in O(log n) each. If the data will be searched only once, linear search wins because sorting first costs more than the search itself.

Binary search is notoriously easy to get subtly wrong. The classic bugs are off-by-one on the bounds (hi = len(arr) vs hi = len(arr) - 1), the wrong loop condition (< vs <=), and the wrong midpoint update (mid vs mid + 1). The version in the topic code uses the inclusive convention: hi is the last valid index, the loop runs while lo <= hi, and after a miss the new bound is one past mid. Stick with one convention and apply it consistently.

  • Linear (sequential) search — O(n), no preconditions.
  • Binary search — O(log n), requires sorted input.
  • Precondition — a property assumed true at the start of an algorithm.
  • Loop invariant — a property true at the start of each iteration.
  • Worst / best / average case — bounds on running time for the worst, best, and typical inputs.
  • Divide-and-conquer — split the problem into smaller sub-problems and combine.

Worked example — tracing binary search

arr = [3, 7, 12, 19, 24, 31, 42, 58, 66, 71]
target = 31

# lo=0  hi=9  mid=4  arr[4]=24 < 31  → lo=5
# lo=5  hi=9  mid=7  arr[7]=58 > 31  → hi=6
# lo=5  hi=6  mid=5  arr[5]=31 == 31 → return 5

Three comparisons locate the target in a 10-element list. In general the loop runs at most ⌊log₂ n⌋ + 1 times, because each iteration halves the size of the search window. The invariant is: if the target is in the array, it lies somewhere in arr[lo..hi]. When lo > hi the invariant proves the target is absent and the function returns -1.

Worked example — when not to use binary search

# Need to search a list 5 times
data = read_unsorted_list()      # n items
# Linear: 5 × O(n) = O(5n)
# Sort + binary: O(n log n) + 5 × O(log n) = O(n log n)
# Linear wins when 5n < n log n, i.e. n < 32

Algorithm selection is not just "binary is faster". For small n, or few queries, the constants of sorting dominate. Big-O describes asymptotic growth; pragmatic choice still depends on concrete sizes.

  • Running binary search on an unsorted list — silently wrong answers.
  • Off-by-one on bounds: hi = len(arr) with loop <= overruns by one.
  • Forgetting to update lo or hi inside the loop → infinite loop.
  • Computing mid = (lo + hi) / 2 in languages where this overflows for large values (Python is safe; Java/C are not — use lo + (hi - lo) // 2).
  • Returning the wrong "not found" sentinel — some APIs prefer None over -1.

4.2 — Bubble, Insertion, and Selection Sort
The three classical O(n²) sorts; correctness invariants; in-place vs. stable sort
def insertion_sort(a): for i in range(1, len(a)): key = a[i] j = i - 1 while j >= 0 and a[j] > key: a[j+1] = a[j] j -= 1 a[j+1] = key return a
📖 Lesson Notes — Ontario Curriculum

The three O(n²) sorts — bubble, insertion, selection — are studied not because anyone uses them on large data sets but because they illustrate algorithmic concepts cleanly. Each maintains a different loop invariant, and each makes different trade-offs in number of comparisons, number of swaps, and adaptability to already-sorted input. Understanding these baselines makes the O(n log n) sorts in the next topic genuinely impressive.

Bubble sort repeatedly walks the array, swapping adjacent out-of-order pairs. After pass k, the k largest elements are in their final positions at the end. Worst case O(n²), best case O(n) on an already-sorted array if the implementation tracks "no swaps this pass". Selection sort finds the minimum of the unsorted portion and swaps it to the front; it always makes exactly n - 1 swaps regardless of input, with O(n²) comparisons. Insertion sort builds the sorted prefix one element at a time, shifting elements right to make room — best case O(n) on sorted input, worst case O(n²), and an excellent choice for small or nearly-sorted arrays.

A sort is stable if equal-keyed elements keep their original relative order. Bubble and insertion sort are stable; selection sort, naively implemented, is not. A sort is in-place if it uses O(1) extra memory; all three are. These properties matter when sorting records by one field while preserving order on another.

  • Comparison sort — sorts by comparing pairs of elements; lower bound O(n log n).
  • Stable sort — equal keys preserve original order.
  • In-place sort — uses O(1) auxiliary memory.
  • Adaptive sort — runs faster on already-near-sorted input.
  • Inversion — a pair (i, j) with i < j but a[i] > a[j]; insertion sort runs in O(n + #inversions).
  • Pass — one complete sweep through the array.

Worked example — selection sort with trace

def selection_sort(a):
    for i in range(len(a) - 1):
        # invariant: a[0..i-1] is sorted and holds the i smallest
        min_idx = i
        for j in range(i + 1, len(a)):
            if a[j] < a[min_idx]:
                min_idx = j
        a[i], a[min_idx] = a[min_idx], a[i]
    return a

# Trace on [5, 2, 8, 1]
# i=0: find min in [5,2,8,1] → 1 at idx 3 → swap → [1,2,8,5]
# i=1: find min in [2,8,5]   → 2 at idx 1 → swap with self → [1,2,8,5]
# i=2: find min in [8,5]     → 5 at idx 3 → swap → [1,2,5,8]
# done

The outer loop runs n - 1 times; the inner loop runs n - i - 1 times. Total comparisons ≈ n(n-1)/2 = O(n²) regardless of input order. Only n - 1 swaps are performed, which is the minimum possible — useful when swaps are far more expensive than comparisons (e.g., swapping large records or values in slow storage).

  • Confusing the "current minimum index" pattern in selection sort with the "shift right" pattern in insertion sort.
  • Bubble-sort implementations that always run n full passes, missing the early-exit optimisation that makes the best case O(n).
  • Off-by-one in the inner loop bound — running past the end with j+1.
  • Believing selection sort is stable — it is not without a special tie-break.
  • Sorting a list by repeatedly calling Python's sorted inside a loop — turns an O(n log n) job into O(n² log n).

4.3 — Merge Sort and Quicksort (Divide & Conquer)
Recursive sort algorithms; merge step; pivot selection; average vs. worst-case behavior
def merge_sort(a): if len(a) <= 1: return a mid = len(a) // 2 left = merge_sort(a[:mid]) right = merge_sort(a[mid:]) # merge out, i, j = [], 0, 0 while i < len(left) and j < len(right): if left[i] <= right[j]: out.append(left[i]); i += 1 else: out.append(right[j]); j += 1 out += left[i:] + right[j:] return out
📖 Lesson Notes — Ontario Curriculum

Merge sort and quicksort are the two flagship O(n log n) sorting algorithms, and both use the divide-and-conquer paradigm. Merge sort recursively splits the array in half, sorts each half, then merges the sorted halves; quicksort picks a pivot, partitions the array into "less than pivot" and "greater than pivot", and recurses on each part. The two have the same average complexity but very different practical behaviour, which is why both are still taught and still used.

Merge sort guarantees O(n log n) in every case: best, average, and worst. The recursion tree has log₂ n levels and each level does O(n) work merging. It is also stable, which matters when sorting records on multiple keys. Its drawback is that it needs O(n) extra memory for the merge step, ruling it out for memory-constrained environments. Quicksort averages O(n log n) and runs slightly faster in practice because of cache-friendly access patterns and in-place partitioning. But its worst case is O(n²), which occurs when the pivot is consistently the smallest or largest element — for example on already-sorted input with naive "first element" pivoting. Real-world quicksort implementations choose the pivot carefully (random, median-of-three) and fall back to insertion sort or heapsort to guarantee worst-case behaviour (introsort).

Python's built-in sorted uses Timsort, a hybrid of merge sort and insertion sort that exploits naturally occurring "runs" in real-world data and is O(n) on already-sorted input. The lesson is that idealised textbook sorts are starting points; production sorts are engineered.

  • Divide and conquer — split, solve recursively, combine.
  • Merge step — combining two sorted lists into one sorted list in O(n).
  • Pivot — the element around which quicksort partitions.
  • Partition — re-arranging so that left ≤ pivot ≤ right.
  • Stable / in-place — same definitions as in the O(n²) sorts.
  • Recurrence relation — T(n) = 2·T(n/2) + O(n) for merge sort, solving to O(n log n) by the Master Theorem.

Worked example — quicksort with Lomuto partition

def quicksort(a, lo=0, hi=None):
    if hi is None: hi = len(a) - 1
    if lo < hi:
        p = partition(a, lo, hi)
        quicksort(a, lo, p - 1)
        quicksort(a, p + 1, hi)
    return a

def partition(a, lo, hi):
    pivot = a[hi]              # last element as pivot
    i = lo - 1
    for j in range(lo, hi):
        if a[j] <= pivot:
            i += 1
            a[i], a[j] = a[j], a[i]
    a[i+1], a[hi] = a[hi], a[i+1]
    return i + 1

print(quicksort([5, 2, 8, 1, 9, 3]))  # [1, 2, 3, 5, 8, 9]

Average case: each partition splits roughly in half, giving log n levels of recursion with O(n) partition work per level → O(n log n). Worst case (sorted input, last-element pivot): every partition produces a 0/n-1 split, giving n levels → O(n²). Picking a random pivot or the median-of-three makes the worst case vanishingly unlikely in practice. The algorithm is in-place (O(log n) stack space on average) but not stable.

  • Using deterministic pivot on adversarial input — degrades to O(n²).
  • Forgetting that the merge step in textbook merge sort uses O(n) extra memory.
  • Naive quicksort recursing without tail-recursion elimination on huge arrays — stack overflow.
  • Hand-implementing sorts in production code instead of using sorted() / Arrays.sort().
  • Forgetting that sorted(lst) returns a new list; lst.sort() mutates in place.

4.4 — Big-O Notation and Algorithm Efficiency
O(1), O(log n), O(n), O(n log n), O(n²), O(2ⁿ); growth rates; counting basic operations
📖 Lesson Notes — Ontario Curriculum

Big-O notation describes how the running time (or memory usage) of an algorithm grows as the input size n grows. It is an asymptotic upper bound: it ignores constant factors and lower-order terms, focusing on the leading behaviour for large n. Saying an algorithm is O(n²) does not mean it takes exactly n² operations — it means there exist constants c and n₀ such that the run time is at most c·n² for all n ≥ n₀. This abstraction is what lets students reason about algorithm efficiency without knowing the speed of the hardware.

The common complexity classes, in increasing order, are O(1) (constant — array indexing, hash lookup), O(log n) (logarithmic — binary search, balanced-tree operations), O(n) (linear — single loop over input), O(n log n) (linearithmic — merge sort, heapsort), O(n²) (quadratic — nested loops, bubble sort), O(n³) (cubic — naive matrix multiplication), and O(2ⁿ) (exponential — brute-force subset enumeration). The gap between these classes is enormous: for n = 1,000,000, log n ≈ 20, n log n ≈ 2 × 10⁷, and n² ≈ 10¹², which on a 10⁹-operations-per-second machine takes roughly 20 microseconds vs 20 milliseconds vs over 16 minutes.

To analyse code, count the dominant basic operations as a function of n. Nested loops multiply; sequential blocks add (and the larger dominates); recursive calls follow a recurrence solved by the Master Theorem or by drawing the recursion tree. Constants and lower-order terms are dropped: O(3n + 7) = O(n), O(n² + n) = O(n²).

  • Big-O — asymptotic upper bound.
  • Big-Ω (Omega) — asymptotic lower bound.
  • Big-Θ (Theta) — both bounds (tight estimate).
  • Best / average / worst case — bounds over best, typical, and worst inputs.
  • Amortised complexity — average cost per operation over a long sequence (e.g., Python list .append is amortised O(1)).
  • Space complexity — same idea applied to memory rather than time.
  • Polynomial vs exponential — the boundary between "tractable" and "intractable".

Worked example — analysing four code fragments

# (1) O(1)
def first(a):
    return a[0]

# (2) O(n)
def max_val(a):
    m = a[0]
    for x in a:    # n iterations
        if x > m: m = x
    return m

# (3) O(n²)
def has_dup(a):
    for i in range(len(a)):
        for j in range(i+1, len(a)):
            if a[i] == a[j]:
                return True
    return False

# (4) O(log n)
def count_halvings(n):
    c = 0
    while n > 1:
        n //= 2
        c += 1
    return c

(1) Does one indexing operation → constant time. (2) One pass through the list with constant work per element → linear. (3) Each i pairs with up to n-i-1 values of j; total ≈ n²/2 → quadratic. (4) The loop variable halves each iteration; the loop body runs log₂ n times → logarithmic. A faster O(n) alternative for (3) uses a hash set to detect duplicates in one pass — a classic time/space trade-off.

  • Confusing O(n²) with O(2ⁿ) — quadratic vs exponential are vastly different.
  • Including constants: "O(2n)" is just O(n); never write O(½n²) — it's O(n²).
  • Comparing constant-factor-improved versions of the same big-O — both are still O(n).
  • Forgetting hidden costs: building a string with += n times is O(n²), not O(n).
  • Confusing best, average, and worst case — quicksort is O(n log n) average, O(n²) worst.
  • Assuming O(log n) means "with logarithmic operations"; it means the count of operations grows logarithmically.

4.5 — Algorithm Selection and Trade-offs
Choose the right algorithm: small lists, nearly-sorted data, memory constraints, stability requirements
📖 Lesson Notes — Ontario Curriculum

An algorithm's asymptotic complexity is one of several factors that go into choosing it for a given problem. Big-O captures the dominant term for large n, but real-world choices also depend on the size of n, the distribution of the input, the stability required of the result, the memory available, and the cost of operations the abstract model considers "constant" (a memory cache miss can be 100× a register operation). A working programmer must reason about all of these together.

For sorting specifically, the textbook hierarchy translates into the following practical rules. On small inputs (n < ~50), insertion sort beats quicksort because its constant factors are tiny — this is why library sorts switch to insertion sort on small partitions. On nearly-sorted input, insertion sort and Timsort run in O(n); naive quicksort with a fixed pivot degrades to O(n²). With memory pressure, heapsort or in-place quicksort beat merge sort because the latter needs O(n) extra space. When stability is required (e.g., sorting students by GPA while preserving alphabetical order on ties), merge sort and Timsort qualify; heapsort and naive quicksort do not. External sorting on data larger than RAM uses merge-sort-style passes over disk because random access is too expensive.

Beyond sorting, algorithm selection covers the search-vs-hash trade-off (O(log n) tree vs O(1) hash, the former giving order, the latter not), the iterative-vs-recursive trade-off (stack depth, code clarity), and the precompute-vs-compute trade-off (memoise expensive sub-results once and reuse). Asking what trade-offs am I making? is the mark of a Grade 12-level programmer.

  • Constant factors — the hidden multipliers that Big-O ignores but real time depends on.
  • Adaptive algorithm — one whose run time improves on favourable input.
  • Stability — preserves the relative order of equal-keyed elements.
  • In-place — uses O(1) auxiliary memory.
  • External algorithm — designed for data that does not fit in RAM.
  • Trade-off — improving one resource at the cost of another (time vs space, simplicity vs speed).

Worked example — pick the right tool

# Scenario A: 30 student marks, sort by mark
# → insertion sort or built-in sorted() — both fine, tiny n

# Scenario B: 10 million log lines, sort by timestamp
# → built-in sorted() (Timsort), or external merge sort if RAM-limited

# Scenario C: sort employees by salary, preserve original
#             order on ties for fairness in tie-breaks
# → MUST be stable: merge sort, Timsort; NOT heapsort

# Scenario D: maintain a "top 10" list as new scores arrive
# → don't sort each time; use a min-heap of size 10 (O(log 10) per insert)

Scenario D illustrates a common pattern: choosing an entirely different data structure (a heap) avoids the sorting problem altogether. Time per insertion is O(log 10) = O(1) constant — the heap maintains its invariant incrementally, so the running cost of n insertions is O(n) rather than the O(n²) of "sort after each insert" or even the O(n · n log n) of "re-sort each time".

  • Picking the asymptotically best algorithm without checking whether n is large enough for it to matter.
  • Ignoring memory: a great O(n log n) sort that needs 16 GB of RAM is useless on a 4 GB device.
  • Forgetting stability when sorting tabular data on multiple keys.
  • Re-running an O(n log n) sort every time one element is added, when an O(log n) heap insert suffices.
  • Choosing recursion for problems with deep call chains and no tail-call optimisation.

5.1 — Testing & Debugging Strategies
Unit tests, boundary cases, regression testing, debugger basics, the rubber-duck method, common bug types (off-by-one, type, scope)
import unittest def divide(a, b): if b == 0: raise ValueError("divide by zero") return a / b class TestDivide(unittest.TestCase): def test_basic(self): self.assertEqual(divide(10, 2), 5) def test_zero(self): self.assertRaises(ValueError, divide, 10, 0)
📖 Lesson Notes — Ontario Curriculum

Testing verifies that a program does what it is supposed to do; debugging is the process of locating and fixing the cause when it does not. The two go hand in hand: a good test suite makes debugging easier because failing tests point precisely at what broke. Professional development teams write unit tests alongside the code they exercise — small, fast tests that examine one function at a time, asserting that specific inputs produce specific outputs.

The test-pyramid view of testing has many unit tests at the base (fast, focused), fewer integration tests in the middle (groups of modules working together), and a handful of end-to-end tests at the top (the whole system). When designing test cases, cover typical inputs, boundary inputs (empty list, single element, maximum size), and invalid inputs (negative numbers where positive expected, malformed strings). The boundary is where most bugs hide.

Debugging tactics include print-debugging (sprinkling temporary print statements to trace state), using an interactive debugger with breakpoints and single-stepping, reading the stack trace from top to bottom to find your own code in the call chain, the rubber-duck method (explaining the code aloud to an inanimate object often reveals the bug), and bisection (commenting out half the code to localise the failure). Regression tests — tests added after a bug is fixed — prevent the same bug from re-appearing.

  • Unit test — exercises a single function or class.
  • Integration test — exercises multiple modules together.
  • Assertion — assertEqual, assertTrue, etc., that fails the test if the condition is false.
  • Fixture — set-up and tear-down code shared by multiple tests.
  • Boundary case — input at the edge of valid range.
  • Regression — a previously fixed bug that re-appears.
  • Test coverage — proportion of code exercised by tests.
  • Stack trace — the chain of function calls active when an exception was raised.

Worked example — designing a thorough test suite

import unittest

def median(values):
    """Return the median of a non-empty list of numbers."""
    s = sorted(values)
    n = len(s)
    if n == 0:
        raise ValueError("median of empty list")
    mid = n // 2
    if n % 2 == 1:
        return s[mid]
    return (s[mid-1] + s[mid]) / 2

class TestMedian(unittest.TestCase):
    def test_single(self):       self.assertEqual(median([7]), 7)
    def test_odd_length(self):   self.assertEqual(median([1, 3, 2]), 2)
    def test_even_length(self):  self.assertEqual(median([1, 2, 3, 4]), 2.5)
    def test_unsorted_input(self):
        self.assertEqual(median([5, 1, 4, 2, 3]), 3)
    def test_negatives(self):    self.assertEqual(median([-2, -1, 0]), -1)
    def test_duplicates(self):   self.assertEqual(median([2, 2, 2]), 2)
    def test_empty_raises(self):
        with self.assertRaises(ValueError):
            median([])

Seven tests cover: single element (boundary), odd/even length (parity), unsorted (the function must sort internally), negatives, duplicates, and empty (error path). Each test runs in O(n log n) but n is tiny in tests. The whole suite finishes in milliseconds and gives confidence that future changes won't silently break median.

  • Testing only happy-path inputs — bugs live in the edge cases.
  • Writing tests that all pass because the assertions are too weak (assertTrue(result) instead of assertEqual(result, 42)).
  • Print-debugging then forgetting to remove the prints before committing.
  • Modifying code without running existing tests — invites regressions.
  • Ignoring the first failing test and skipping ahead — fix in order; later failures may be downstream effects.

5.2 — Version Control with Git (Essentials)
Repos, commits, branches, merges, pull requests, .gitignore; why version control matters in collaborative software
📖 Lesson Notes — Ontario Curriculum

A version control system (VCS) is a tool that records the complete history of changes to a code base. Every modern software team uses one, and Git dominates the field. The mental model is simple: a Git repository stores a directed graph of commits, where each commit is a snapshot of the project plus a pointer to its parent. A branch is just a movable label pointing at the tip of a line of commits; creating one is cheap and instant. Merging combines two branches' histories, producing a new commit with two parents.

Why does this matter? First, version control is the undo button for software — any earlier state can be restored, any change reviewed, any author identified. Second, it enables parallel work: two developers can work on different branches, then merge, with the VCS handling most of the bookkeeping. Third, it enables code review: changes are proposed as pull requests, examined by teammates, and only merged when approved. Fourth, it provides traceability: every line of code can be traced back to the commit that introduced it (git blame), which is invaluable for bug archaeology.

The discipline of good commits is more important than knowing every Git command. A commit should represent one logical change, with a message that says what and why (not how — the diff already shows how). The .gitignore file lists artefacts that should not be tracked: compiled files, virtual environments, secrets, IDE configuration. Committing a file that should not be tracked (especially API keys or passwords) is a security incident; the history makes it hard to fully remove.

  • Repository (repo) — the project's history and metadata.
  • Commit — a snapshot plus parent pointer; identified by a SHA-1 hash.
  • Branch — a movable pointer to a commit, enabling parallel histories.
  • Merge — combining two branches into one.
  • Pull request (PR) / merge request (MR) — proposing a branch for review.
  • Remote — a copy of the repo hosted elsewhere (often on GitHub or GitLab).
  • .gitignore — list of files Git should not track.
  • Merge conflict — when two branches change the same lines and Git can't auto-merge.

Worked example — a typical Git workflow

# Start a new feature
git checkout -b feature/letter-grades

# Work, then stage and commit
git add grader.py tests/test_grader.py
git commit -m "Add letter-grade conversion using Ontario achievement chart"

# Push to remote and open a pull request
git push -u origin feature/letter-grades

# After review, merge back into main
git checkout main
git pull
git merge feature/letter-grades
git push

Each step has a precise effect on the graph: checkout -b creates a new branch pointer; add stages changes for the next commit; commit creates the snapshot; push uploads commits to the remote; merge creates the merge commit. Mistakes can almost always be recovered because Git rarely deletes anything — even after a "delete" the SHA usually still references a reachable object for a while.

  • Committing secrets or .env files — anyone who later clones the repo can read them.
  • Massive omnibus commits ("fix everything") — impossible to review or revert.
  • Working directly on main instead of a branch — increases risk of breaking shared state.
  • Forcing pushes (git push --force) on shared branches — rewrites history teammates rely on.
  • Vague commit messages like "stuff" or "wip" — useless three weeks later.
  • Failing to pull before pushing — leads to unnecessary merge conflicts.

5.3 — The Software Development Life Cycle & Maintainability
Requirements gathering, design, implement, test, deploy, maintain; commenting; naming conventions (PEP 8); code reviews
📖 Lesson Notes — Ontario Curriculum

The Software Development Life Cycle (SDLC) is the framework of activities that turn a problem statement into running, maintained software. The classical stages are requirements analysis (what does the user actually need?), design (how will the system be structured?), implementation (writing the code), testing (verifying behaviour), deployment (releasing to users), and maintenance (fixing bugs and adapting to change). Empirical studies show maintenance consumes 60-80% of total software cost — so writing code that can be maintained is at least as important as making it work the first time.

SDLC methodologies organise these stages differently. The waterfall model runs them sequentially; it is rigid but well suited to safety-critical systems with stable requirements. Agile (and its popular variant Scrum) iterates through all stages in short sprints (typically 1-4 weeks), producing a working increment each cycle and welcoming changing requirements. Most modern teams use some form of Agile because real users rarely know exactly what they want until they see something working.

Code quality has measurable indicators: readability, cohesion, coupling, cyclomatic complexity (the number of independent paths through a function — lower is better). Style guides such as PEP 8 (Python) and Google Java Style codify naming, indentation, line length, and comment conventions so that one engineer's code looks like another's. Code review — a second pair of eyes on every change — is the single most effective quality practice in industry, catching bugs and spreading knowledge simultaneously.

  • SDLC — sequence of activities from problem to deployed software.
  • Waterfall — strict sequential stages.
  • Agile / Scrum — short iterations with frequent user feedback.
  • Requirement — a statement of what the software must do (functional) or how well (non-functional).
  • Maintainability — the ease of changing software after release.
  • Style guide — coding conventions (e.g. PEP 8 for Python).
  • Code review — peer evaluation of changes before they are merged.
  • Technical debt — accumulated short-cuts that slow future development.

Worked example — refactoring for maintainability

# BEFORE — undocumented, magic numbers, poor names
def f(x):
    if x >= 80: return "A"
    if x >= 70: return "B"
    if x >= 60: return "C"
    if x >= 50: return "D"
    return "F"

# AFTER — named constants, docstring, type hints, tested
LEVEL_4_THRESHOLD = 80   # Ontario Level 4: 80-100%
LEVEL_3_THRESHOLD = 70   # Ontario Level 3: 70-79%
LEVEL_2_THRESHOLD = 60   # Ontario Level 2: 60-69%
LEVEL_1_THRESHOLD = 50   # Ontario Level 1: 50-59%

def percent_to_letter(percent: float) -> str:
    """Map an Ontario achievement-chart percent to a letter grade.

    Args:
        percent: 0-100 inclusive.
    Returns:
        'A' for Level 4, 'B' for Level 3, …, 'F' for below 50.
    """
    if percent >= LEVEL_4_THRESHOLD: return "A"
    if percent >= LEVEL_3_THRESHOLD: return "B"
    if percent >= LEVEL_2_THRESHOLD: return "C"
    if percent >= LEVEL_1_THRESHOLD: return "D"
    return "F"

The two versions behave identically — both are O(1) — but the second can be understood, modified, and tested by someone other than the original author. The named thresholds prevent the "magic number" anti-pattern. The docstring documents the contract. Type hints let editors and static checkers flag misuse. This level of care is what makes a codebase maintainable over years.

  • Implementing before talking to users — building the wrong thing perfectly.
  • Skipping the design step on a "small" project that grows — costs accumulate as technical debt.
  • Commenting what the code does instead of why — comments duplicate code and rot.
  • Inconsistent naming (my_var, thatThing, X) — slows every future reader.
  • Treating code review as approval theatre rather than substantive discussion.
  • "It works on my machine" — different environments need explicit dependency management.

5.4 — Emerging Technologies: AI, Machine Learning, Quantum Computing
High-level overview of how supervised learning, neural networks, and large-language-models work; introduction to qubits and quantum advantage
📖 Lesson Notes — Ontario Curriculum

Artificial intelligence (AI) is the broad ambition of building systems that perform tasks normally requiring human intelligence. Machine learning (ML) is the dominant modern approach: rather than programming explicit rules, the developer supplies data and the algorithm discovers patterns. In supervised learning, training data is paired with correct labels, and the model learns to map inputs to outputs that generalise to new examples. Unsupervised learning works with unlabelled data, finding structure (clusters, anomalies). Reinforcement learning trains an agent by reward signals from interaction with an environment.

Neural networks are the dominant model family. Each artificial neuron computes a weighted sum of its inputs, applies a non-linear activation function, and passes the result on. Stacked into deep networks with millions or billions of trainable weights, they can approximate complex functions. Large language models (LLMs) like the one generating this text are transformer-based neural networks trained on enormous text corpora to predict the next token; emergent abilities like question answering and translation arise from scale and architecture, not from any explicit grammar programmed in.

Quantum computing exploits superposition and entanglement of quantum bits (qubits) to evaluate exponentially many possibilities simultaneously. Shor's algorithm factors integers in polynomial time on a quantum computer, which would break RSA encryption; Grover's algorithm searches unsorted data in O(√n). As of the late 2020s, quantum hardware has reached "noisy intermediate-scale" capability — useful for research but not yet for general-purpose computing. Quantum-resistant cryptography (post-quantum) is already a topic of standards work at NIST.

  • Training data — examples used to fit a model's parameters.
  • Feature — one measurable property of the input.
  • Model — a mathematical function with learnable parameters.
  • Overfitting — performing well on training data but poorly on new data.
  • Generalisation — performing well on unseen inputs.
  • Neural network — a graph of weighted connections inspired (loosely) by biology.
  • Token — the basic unit (word/sub-word) an LLM processes.
  • Qubit — a quantum two-state system; can be in superposition of 0 and 1.
  • Entanglement — correlated quantum state of multiple qubits.

Worked example — a tiny k-NN classifier

def euclidean(a, b):
    return sum((x - y) ** 2 for x, y in zip(a, b)) ** 0.5

def knn_predict(train, labels, query, k=3):
    """Predict label of `query` using k-nearest neighbours."""
    distances = [(euclidean(point, query), lbl)
                 for point, lbl in zip(train, labels)]
    distances.sort()                       # O(n log n)
    top_k = distances[:k]
    votes = {}
    for _, lbl in top_k:
        votes[lbl] = votes.get(lbl, 0) + 1
    return max(votes, key=votes.get)       # most common label

train  = [[1,2],[2,1],[5,5],[6,6]]
labels = ['cat','cat','dog','dog']
print(knn_predict(train, labels, [2,2]))   # 'cat'

k-NN is the gentlest entry into machine learning: there is no "training" beyond storing the data, and prediction is a sort plus a vote. With n training points and d features, prediction costs O(n·d + n log n) per query — fine for thousands of points, infeasible for millions without approximate-NN data structures. It illustrates the central ML idea: classification by similarity to known examples, parameterised by the choice of distance metric and k.

  • Believing ML "understands" — current models are statistical pattern matchers and can be confidently wrong (hallucinations).
  • Training and evaluating on the same data — gives an inflated, useless accuracy score.
  • Ignoring bias in training data — models inherit the patterns, including discriminatory ones.
  • Conflating quantum computing with "faster classical computing" — it accelerates specific problem classes only.
  • Assuming LLM outputs are factual — they predict plausible text, not truth.

5.5 — Ethical & Environmental Issues in Computing
Privacy, algorithmic bias, intellectual property, e-waste, energy footprint of data centres & AI training, ethical AI design
📖 Lesson Notes — Ontario Curriculum

Computing technology shapes the lives of billions of people, which makes its ethical and environmental dimensions an inseparable part of the discipline. Software engineers are no longer simply "doing what the spec says" — the decisions they make about data, algorithms, and deployment have consequences for privacy, fairness, employment, the environment, and democratic discourse. Recognising the moral weight of these decisions is part of being a professional.

Privacy is the foundational issue. Modern applications collect quantities of personal data that would have been unthinkable a generation ago. Canadian law (PIPEDA at the federal level; provincial laws in Quebec, Alberta, and BC) requires informed consent and limited retention. Designers should follow the data minimisation principle: collect only what is needed, store it for only as long as needed, and protect it (encryption at rest and in transit) for the entire time. Algorithmic bias occurs when models trained on historical data perpetuate or amplify existing inequities — credit-scoring systems, hiring filters, and predictive-policing tools have all been shown to discriminate. Mitigations include diverse training data, fairness metrics, human-in-the-loop review, and the right to contest automated decisions.

Intellectual property protects the work of programmers and content creators. Copyright covers expressive works (including source code) automatically. Open-source licences (MIT, GPL, Apache 2) grant explicit permissions in exchange for conditions like attribution or copyleft. Using code without understanding its licence is a real legal risk. The training of large generative models on copyrighted data is an active legal frontier.

Environmental impact is rising fast. Data-centre electricity consumption is projected to exceed 4% of global use by the late 2020s; training a single large language model can emit as much CO₂ as several transatlantic flights; the production of electronics is water-intensive and rare-earth-mineral-intensive. The end of a device's life — e-waste — accounts for the fastest-growing solid-waste stream worldwide. Mitigations include energy-efficient algorithms, sourcing renewable power, designing for repair and modularity, and extending device lifespans through software support.

  • PIPEDA — Personal Information Protection and Electronic Documents Act (Canadian federal privacy law).
  • Data minimisation — collect only what is necessary.
  • Encryption — converting data so only authorised parties can read it.
  • Algorithmic bias — unfair outcomes from biased training data or design.
  • Open-source licence — legal terms for reuse of code (MIT, GPL, Apache 2, BSD).
  • Copyleft — licence clause requiring derivative works to use the same licence.
  • E-waste — discarded electronics; the largest growing waste stream.
  • Carbon footprint — total greenhouse gas emissions attributable to an activity.

Worked example — designing a school attendance app responsibly

Suppose your team must build an attendance-tracking app for a Toronto-area school board. An ethical design checklist:

  • Minimise data — store student ID, date, status (present/late/absent). Do not store GPS, photos, or browsing.
  • Retention — purge records after the legally required retention period (typically 5 years for Ontario school records).
  • Encryption — HTTPS in transit; encrypt the database at rest.
  • Access control — students see their own records; teachers see their class; administrators see the school. No global access without audit.
  • Bias — if attendance flags trigger interventions, ensure thresholds do not disproportionately catch students from particular demographics; build in human review.
  • Sustainability — host on a green data-centre region; cache aggressively to reduce server cycles.
  • Transparency — provide a privacy notice in plain language; let students export and delete their data.

None of these decisions is purely technical; every one balances utility, privacy, fairness, and cost. Making them deliberately, rather than by default, is the ethical commitment.

  • Storing personal data "just in case" — accumulates risk with no benefit.
  • Logging passwords, tokens, or full credit card numbers — a future leak becomes catastrophic.
  • Treating "the algorithm decided" as a defence against discrimination — algorithms reflect their training.
  • Copying code from a Stack Overflow answer without checking the licence — Stack Overflow content is CC BY-SA, requiring attribution.
  • Designing for the dominant user group only — accessibility and inclusion are quality attributes, not nice-to-haves.
  • Ignoring the carbon cost of model training and large-scale data movement.

5.6 — History of Computing & Career Pathways
From Babbage and Lovelace to modern cloud computing; CS-related careers in Canada (software dev, data science, security, ML, robotics, game dev); Canadian post-secondary CS programs
📖 Lesson Notes — Ontario Curriculum

Computer science is unusual among disciplines in that its entire intellectual history fits within roughly two centuries — and most of it within one human lifetime. Charles Babbage designed the Analytical Engine in the 1830s and Ada Lovelace wrote the first algorithm intended for machine execution, earning her the title of the first programmer. Alan Turing's 1936 paper defined the abstract machine that bears his name and proved the existence of uncomputable problems. The ENIAC (1945) was the first general-purpose electronic computer; the transistor (1947), integrated circuit (1958), and microprocessor (1971) drove exponential miniaturisation captured by Moore's Law. The personal-computer era (1977 onward), the internet (1980s research, 1990s commercial), and the smartphone (2007 iPhone) each shifted computing's role in society dramatically. The current era is defined by cloud computing (utility-priced compute and storage), mobile-first design, and generative AI.

Canada plays a distinctive role: Geoffrey Hinton, Yoshua Bengio, and Yann LeCun (the "godfathers of deep learning") all studied or worked in Canadian universities, and Toronto and Montreal remain global hubs for AI research. The Vector Institute (Toronto), Mila (Montreal), and Amii (Edmonton) are world-class research centres. Major Canadian post-secondary computer-science programs include Waterloo, Toronto, McGill, UBC, McMaster, Western, Queen's, Carleton, Ottawa, Simon Fraser, Alberta, and the polytechnic colleges (Sheridan, Seneca, BCIT) for applied programs.

Career paths in computing are now extremely diverse. Software developer remains the most common role; specialisations include front-end (browser UI), back-end (server APIs), full-stack (both), mobile, game development, and embedded systems. Data scientist and machine-learning engineer roles apply statistics and ML to business problems. Cybersecurity analysts defend systems against attack. DevOps and site reliability engineers run infrastructure. Cloud architects, database administrators, QA engineers, technical writers, product managers, and UX designers round out a typical software organisation. Many of these careers do not require a four-year university degree — community college programs and self-directed pathways are increasingly viable.

  • Turing machine — abstract model of computation that defines what is computable.
  • Moore's Law — observation that transistor counts double roughly every 2 years.
  • Cloud computing — on-demand delivery of compute and storage over the internet.
  • DevOps — culture and tools that combine development and operations.
  • SRE (Site Reliability Engineering) — software-engineering approach to operations.
  • FAANG / MAANG — large US tech firms; Canadian equivalents include Shopify, OpenText, Constellation, BlackBerry.
  • Co-op / internship — paid work term integrated with study (Waterloo's program is world-famous).
  • Open source contribution — public portfolio that demonstrates skills to employers.

Worked example — a career-planning checklist for an Ontario Grade 12 student

  • Foundations — solid grasp of one mainstream language (Python or Java), data structures, and Git. ICS4U covers all three.
  • Math — calculus and discrete math help in CS programs; statistics is essential for data roles.
  • Public portfolio — a GitHub profile with three or four substantive projects beats a résumé alone.
  • Choose an entry point — university (broad theory + research access), polytechnic college (faster, applied), bootcamp (intensive, networking-oriented), self-study (cheapest but requires discipline).
  • Apply early — Waterloo, Toronto, and McGill CS admissions are highly competitive; OUAC applications open in fall of Grade 12.
  • Co-op — programs like Waterloo CS or Toronto's PEY interleave paid work terms with study; many graduates have 2 years of industry experience at convocation.
  • Stay learning — technology shifts every 5-10 years; the durable skills are reading code, debugging, communicating, and learning fast.
  • Believing CS is just "coding" — the field is much broader (architecture, security, theory, HCI, applied math).
  • Picking a university by ranking alone — fit, co-op opportunities, and city matter more for outcomes.
  • Treating new tools as "fashions" to ignore — yesterday's fashion (Git, the cloud, ML) is today's foundation.
  • Underestimating soft skills — code review, technical writing, and collaboration are graded on the job.
  • Assuming the AI revolution makes programmers obsolete — it raises the floor of productivity but still rewards rigour and judgement.
