5.1 — Testing & Debugging Strategies
Unit tests, boundary cases, regression testing, debugger basics, the rubber-duck method, common bug types (off-by-one, type, scope)
import unittest def divide(a, b): if b == 0: raise ValueError("divide by zero") return a / b class TestDivide(unittest.TestCase): def test_basic(self): self.assertEqual(divide(10, 2), 5) def test_zero(self): self.assertRaises(ValueError, divide, 10, 0)
📖 Lesson Notes — Ontario Curriculum

Testing verifies that a program does what it is supposed to do; debugging is the process of locating and fixing the cause when it does not. The two go hand in hand: a good test suite makes debugging easier because failing tests point precisely at what broke. Professional development teams write unit tests alongside the code they exercise — small, fast tests that examine one function at a time, asserting that specific inputs produce specific outputs.

The test-pyramid view of testing has many unit tests at the base (fast, focused), fewer integration tests in the middle (groups of modules working together), and a handful of end-to-end tests at the top (the whole system). When designing test cases, cover typical inputs, boundary inputs (empty list, single element, maximum size), and invalid inputs (negative numbers where positive expected, malformed strings). The boundary is where most bugs hide.

Debugging tactics include print-debugging (sprinkling temporary print statements to trace state), using an interactive debugger with breakpoints and single-stepping, reading the stack trace from top to bottom to find your own code in the call chain, the rubber-duck method (explaining the code aloud to an inanimate object often reveals the bug), and bisection (commenting out half the code to localise the failure). Regression tests — tests added after a bug is fixed — prevent the same bug from re-appearing.

  • Unit test — exercises a single function or class.
  • Integration test — exercises multiple modules together.
  • Assertion — assertEqual, assertTrue, etc., that fails the test if the condition is false.
  • Fixture — set-up and tear-down code shared by multiple tests.
  • Boundary case — input at the edge of valid range.
  • Regression — a previously fixed bug that re-appears.
  • Test coverage — proportion of code exercised by tests.
  • Stack trace — the chain of function calls active when an exception was raised.

Worked example — designing a thorough test suite

import unittest

def median(values):
    """Return the median of a non-empty list of numbers."""
    s = sorted(values)
    n = len(s)
    if n == 0:
        raise ValueError("median of empty list")
    mid = n // 2
    if n % 2 == 1:
        return s[mid]
    return (s[mid-1] + s[mid]) / 2

class TestMedian(unittest.TestCase):
    def test_single(self):       self.assertEqual(median([7]), 7)
    def test_odd_length(self):   self.assertEqual(median([1, 3, 2]), 2)
    def test_even_length(self):  self.assertEqual(median([1, 2, 3, 4]), 2.5)
    def test_unsorted_input(self):
        self.assertEqual(median([5, 1, 4, 2, 3]), 3)
    def test_negatives(self):    self.assertEqual(median([-2, -1, 0]), -1)
    def test_duplicates(self):   self.assertEqual(median([2, 2, 2]), 2)
    def test_empty_raises(self):
        with self.assertRaises(ValueError):
            median([])

Seven tests cover: single element (boundary), odd/even length (parity), unsorted (the function must sort internally), negatives, duplicates, and empty (error path). Each test runs in O(n log n) but n is tiny in tests. The whole suite finishes in milliseconds and gives confidence that future changes won't silently break median.

  • Testing only happy-path inputs — bugs live in the edge cases.
  • Writing tests that all pass because the assertions are too weak (assertTrue(result) instead of assertEqual(result, 42)).
  • Print-debugging then forgetting to remove the prints before committing.
  • Modifying code without running existing tests — invites regressions.
  • Ignoring the first failing test and skipping ahead — fix in order; later failures may be downstream effects.

5.2 — Version Control with Git (Essentials)
Repos, commits, branches, merges, pull requests, .gitignore; why version control matters in collaborative software
📖 Lesson Notes — Ontario Curriculum

A version control system (VCS) is a tool that records the complete history of changes to a code base. Every modern software team uses one, and Git dominates the field. The mental model is simple: a Git repository stores a directed graph of commits, where each commit is a snapshot of the project plus a pointer to its parent. A branch is just a movable label pointing at the tip of a line of commits; creating one is cheap and instant. Merging combines two branches' histories, producing a new commit with two parents.

Why does this matter? First, version control is the undo button for software — any earlier state can be restored, any change reviewed, any author identified. Second, it enables parallel work: two developers can work on different branches, then merge, with the VCS handling most of the bookkeeping. Third, it enables code review: changes are proposed as pull requests, examined by teammates, and only merged when approved. Fourth, it provides traceability: every line of code can be traced back to the commit that introduced it (git blame), which is invaluable for bug archaeology.

The discipline of good commits is more important than knowing every Git command. A commit should represent one logical change, with a message that says what and why (not how — the diff already shows how). The .gitignore file lists artefacts that should not be tracked: compiled files, virtual environments, secrets, IDE configuration. Committing a file that should not be tracked (especially API keys or passwords) is a security incident; the history makes it hard to fully remove.

  • Repository (repo) — the project's history and metadata.
  • Commit — a snapshot plus parent pointer; identified by a SHA-1 hash.
  • Branch — a movable pointer to a commit, enabling parallel histories.
  • Merge — combining two branches into one.
  • Pull request (PR) / merge request (MR) — proposing a branch for review.
  • Remote — a copy of the repo hosted elsewhere (often on GitHub or GitLab).
  • .gitignore — list of files Git should not track.
  • Merge conflict — when two branches change the same lines and Git can't auto-merge.

Worked example — a typical Git workflow

# Start a new feature
git checkout -b feature/letter-grades

# Work, then stage and commit
git add grader.py tests/test_grader.py
git commit -m "Add letter-grade conversion using Ontario achievement chart"

# Push to remote and open a pull request
git push -u origin feature/letter-grades

# After review, merge back into main
git checkout main
git pull
git merge feature/letter-grades
git push

Each step has a precise effect on the graph: checkout -b creates a new branch pointer; add stages changes for the next commit; commit creates the snapshot; push uploads commits to the remote; merge creates the merge commit. Mistakes can almost always be recovered because Git rarely deletes anything — even after a "delete" the SHA usually still references a reachable object for a while.

  • Committing secrets or .env files — anyone who later clones the repo can read them.
  • Massive omnibus commits ("fix everything") — impossible to review or revert.
  • Working directly on main instead of a branch — increases risk of breaking shared state.
  • Forcing pushes (git push --force) on shared branches — rewrites history teammates rely on.
  • Vague commit messages like "stuff" or "wip" — useless three weeks later.
  • Failing to pull before pushing — leads to unnecessary merge conflicts.

Curriculum expectation: Strand B, Expectation B2.4 — "use software project management tools (e.g., version control software, project planning software) to facilitate collaboration on a software project".

5.3 — The Software Development Life Cycle & Maintainability
Requirements gathering, design, implement, test, deploy, maintain; commenting; naming conventions (PEP 8); code reviews
📖 Lesson Notes — Ontario Curriculum

The Software Development Life Cycle (SDLC) is the framework of activities that turn a problem statement into running, maintained software. The classical stages are requirements analysis (what does the user actually need?), design (how will the system be structured?), implementation (writing the code), testing (verifying behaviour), deployment (releasing to users), and maintenance (fixing bugs and adapting to change). Empirical studies show maintenance consumes 60-80% of total software cost — so writing code that can be maintained is at least as important as making it work the first time.

SDLC methodologies organise these stages differently. The waterfall model runs them sequentially; it is rigid but well suited to safety-critical systems with stable requirements. Agile (and its popular variant Scrum) iterates through all stages in short sprints (typically 1-4 weeks), producing a working increment each cycle and welcoming changing requirements. Most modern teams use some form of Agile because real users rarely know exactly what they want until they see something working.

Code quality has measurable indicators: readability, cohesion, coupling, cyclomatic complexity (the number of independent paths through a function — lower is better). Style guides such as PEP 8 (Python) and Google Java Style codify naming, indentation, line length, and comment conventions so that one engineer's code looks like another's. Code review — a second pair of eyes on every change — is the single most effective quality practice in industry, catching bugs and spreading knowledge simultaneously.

  • SDLC — sequence of activities from problem to deployed software.
  • Waterfall — strict sequential stages.
  • Agile / Scrum — short iterations with frequent user feedback.
  • Requirement — a statement of what the software must do (functional) or how well (non-functional).
  • Maintainability — the ease of changing software after release.
  • Style guide — coding conventions (e.g. PEP 8 for Python).
  • Code review — peer evaluation of changes before they are merged.
  • Technical debt — accumulated short-cuts that slow future development.

Worked example — refactoring for maintainability

# BEFORE — undocumented, magic numbers, poor names
def f(x):
    if x >= 80: return "A"
    if x >= 70: return "B"
    if x >= 60: return "C"
    if x >= 50: return "D"
    return "F"

# AFTER — named constants, docstring, type hints, tested
LEVEL_4_THRESHOLD = 80   # Ontario Level 4: 80-100%
LEVEL_3_THRESHOLD = 70   # Ontario Level 3: 70-79%
LEVEL_2_THRESHOLD = 60   # Ontario Level 2: 60-69%
LEVEL_1_THRESHOLD = 50   # Ontario Level 1: 50-59%

def percent_to_letter(percent: float) -> str:
    """Map an Ontario achievement-chart percent to a letter grade.

    Args:
        percent: 0-100 inclusive.
    Returns:
        'A' for Level 4, 'B' for Level 3, …, 'F' for below 50.
    """
    if percent >= LEVEL_4_THRESHOLD: return "A"
    if percent >= LEVEL_3_THRESHOLD: return "B"
    if percent >= LEVEL_2_THRESHOLD: return "C"
    if percent >= LEVEL_1_THRESHOLD: return "D"
    return "F"

The two versions behave identically — both are O(1) — but the second can be understood, modified, and tested by someone other than the original author. The named thresholds prevent the "magic number" anti-pattern. The docstring documents the contract. Type hints let editors and static checkers flag misuse. This level of care is what makes a codebase maintainable over years.

  • Implementing before talking to users — building the wrong thing perfectly.
  • Skipping the design step on a "small" project that grows — costs accumulate as technical debt.
  • Commenting what the code does instead of why — comments duplicate code and rot.
  • Inconsistent naming (my_var, thatThing, X) — slows every future reader.
  • Treating code review as approval theatre rather than substantive discussion.
  • "It works on my machine" — different environments need explicit dependency management.

Curriculum expectation: Strand B, Expectation B2.1 — "describe and use, in computer programs, the standard stages of the software-development life cycle".

5.4 — Emerging Technologies: AI, Machine Learning, Quantum Computing
High-level overview of how supervised learning, neural networks, and large-language-models work; introduction to qubits and quantum advantage
📖 Lesson Notes — Ontario Curriculum

Artificial intelligence (AI) is the broad ambition of building systems that perform tasks normally requiring human intelligence. Machine learning (ML) is the dominant modern approach: rather than programming explicit rules, the developer supplies data and the algorithm discovers patterns. In supervised learning, training data is paired with correct labels, and the model learns to map inputs to outputs that generalise to new examples. Unsupervised learning works with unlabelled data, finding structure (clusters, anomalies). Reinforcement learning trains an agent by reward signals from interaction with an environment.

Neural networks are the dominant model family. Each artificial neuron computes a weighted sum of its inputs, applies a non-linear activation function, and passes the result on. Stacked into deep networks with millions or billions of trainable weights, they can approximate complex functions. Large language models (LLMs) like the one generating this text are transformer-based neural networks trained on enormous text corpora to predict the next token; emergent abilities like question answering and translation arise from scale and architecture, not from any explicit grammar programmed in.

Quantum computing exploits superposition and entanglement of quantum bits (qubits) to evaluate exponentially many possibilities simultaneously. Shor's algorithm factors integers in polynomial time on a quantum computer, which would break RSA encryption; Grover's algorithm searches unsorted data in O(√n). As of the late 2020s, quantum hardware has reached "noisy intermediate-scale" capability — useful for research but not yet for general-purpose computing. Quantum-resistant cryptography (post-quantum) is already a topic of standards work at NIST.

  • Training data — examples used to fit a model's parameters.
  • Feature — one measurable property of the input.
  • Model — a mathematical function with learnable parameters.
  • Overfitting — performing well on training data but poorly on new data.
  • Generalisation — performing well on unseen inputs.
  • Neural network — a graph of weighted connections inspired (loosely) by biology.
  • Token — the basic unit (word/sub-word) an LLM processes.
  • Qubit — a quantum two-state system; can be in superposition of 0 and 1.
  • Entanglement — correlated quantum state of multiple qubits.

Worked example — a tiny k-NN classifier

def euclidean(a, b):
    return sum((x - y) ** 2 for x, y in zip(a, b)) ** 0.5

def knn_predict(train, labels, query, k=3):
    """Predict label of `query` using k-nearest neighbours."""
    distances = [(euclidean(point, query), lbl)
                 for point, lbl in zip(train, labels)]
    distances.sort()                       # O(n log n)
    top_k = distances[:k]
    votes = {}
    for _, lbl in top_k:
        votes[lbl] = votes.get(lbl, 0) + 1
    return max(votes, key=votes.get)       # most common label

train  = [[1,2],[2,1],[5,5],[6,6]]
labels = ['cat','cat','dog','dog']
print(knn_predict(train, labels, [2,2]))   # 'cat'

k-NN is the gentlest entry into machine learning: there is no "training" beyond storing the data, and prediction is a sort plus a vote. With n training points and d features, prediction costs O(n·d + n log n) per query — fine for thousands of points, infeasible for millions without approximate-NN data structures. It illustrates the central ML idea: classification by similarity to known examples, parameterised by the choice of distance metric and k.

  • Believing ML "understands" — current models are statistical pattern matchers and can be confidently wrong (hallucinations).
  • Training and evaluating on the same data — gives an inflated, useless accuracy score.
  • Ignoring bias in training data — models inherit the patterns, including discriminatory ones.
  • Conflating quantum computing with "faster classical computing" — it accelerates specific problem classes only.
  • Assuming LLM outputs are factual — they predict plausible text, not truth.

5.5 — Ethical & Environmental Issues in Computing
Privacy, algorithmic bias, intellectual property, e-waste, energy footprint of data centres & AI training, ethical AI design
📖 Lesson Notes — Ontario Curriculum

Computing technology shapes the lives of billions of people, which makes its ethical and environmental dimensions an inseparable part of the discipline. Software engineers are no longer simply "doing what the spec says" — the decisions they make about data, algorithms, and deployment have consequences for privacy, fairness, employment, the environment, and democratic discourse. Recognising the moral weight of these decisions is part of being a professional.

Privacy is the foundational issue. Modern applications collect quantities of personal data that would have been unthinkable a generation ago. Canadian law (PIPEDA at the federal level; provincial laws in Quebec, Alberta, and BC) requires informed consent and limited retention. Designers should follow the data minimisation principle: collect only what is needed, store it for only as long as needed, and protect it (encryption at rest and in transit) for the entire time. Algorithmic bias occurs when models trained on historical data perpetuate or amplify existing inequities — credit-scoring systems, hiring filters, and predictive-policing tools have all been shown to discriminate. Mitigations include diverse training data, fairness metrics, human-in-the-loop review, and the right to contest automated decisions.

Intellectual property protects the work of programmers and content creators. Copyright covers expressive works (including source code) automatically. Open-source licences (MIT, GPL, Apache 2) grant explicit permissions in exchange for conditions like attribution or copyleft. Using code without understanding its licence is a real legal risk. The training of large generative models on copyrighted data is an active legal frontier.

Environmental impact is rising fast. Data-centre electricity consumption is projected to exceed 4% of global use by the late 2020s; training a single large language model can emit as much CO₂ as several transatlantic flights; the production of electronics is water-intensive and rare-earth-mineral-intensive. The end of a device's life — e-waste — accounts for the fastest-growing solid-waste stream worldwide. Mitigations include energy-efficient algorithms, sourcing renewable power, designing for repair and modularity, and extending device lifespans through software support.

  • PIPEDA — Personal Information Protection and Electronic Documents Act (Canadian federal privacy law).
  • Data minimisation — collect only what is necessary.
  • Encryption — converting data so only authorised parties can read it.
  • Algorithmic bias — unfair outcomes from biased training data or design.
  • Open-source licence — legal terms for reuse of code (MIT, GPL, Apache 2, BSD).
  • Copyleft — licence clause requiring derivative works to use the same licence.
  • E-waste — discarded electronics; the largest growing waste stream.
  • Carbon footprint — total greenhouse gas emissions attributable to an activity.

Worked example — designing a school attendance app responsibly

Suppose your team must build an attendance-tracking app for a Toronto-area school board. An ethical design checklist:

  • Minimise data — store student ID, date, status (present/late/absent). Do not store GPS, photos, or browsing.
  • Retention — purge records after the legally required retention period (typically 5 years for Ontario school records).
  • Encryption — HTTPS in transit; encrypt the database at rest.
  • Access control — students see their own records; teachers see their class; administrators see the school. No global access without audit.
  • Bias — if attendance flags trigger interventions, ensure thresholds do not disproportionately catch students from particular demographics; build in human review.
  • Sustainability — host on a green data-centre region; cache aggressively to reduce server cycles.
  • Transparency — provide a privacy notice in plain language; let students export and delete their data.

None of these decisions is purely technical; every one balances utility, privacy, fairness, and cost. Making them deliberately, rather than by default, is the ethical commitment.

  • Storing personal data "just in case" — accumulates risk with no benefit.
  • Logging passwords, tokens, or full credit card numbers — a future leak becomes catastrophic.
  • Treating "the algorithm decided" as a defence against discrimination — algorithms reflect their training.
  • Copying code from a Stack Overflow answer without checking the licence — Stack Overflow content is CC BY-SA, requiring attribution.
  • Designing for the dominant user group only — accessibility and inclusion are quality attributes, not nice-to-haves.
  • Ignoring the carbon cost of model training and large-scale data movement.

5.6 — History of Computing & Career Pathways
From Babbage and Lovelace to modern cloud computing; CS-related careers in Canada (software dev, data science, security, ML, robotics, game dev); Canadian post-secondary CS programs
📖 Lesson Notes — Ontario Curriculum

Computer science is unusual among disciplines in that its entire intellectual history fits within roughly two centuries — and most of it within one human lifetime. Charles Babbage designed the Analytical Engine in the 1830s and Ada Lovelace wrote the first algorithm intended for machine execution, earning her the title of the first programmer. Alan Turing's 1936 paper defined the abstract machine that bears his name and proved the existence of uncomputable problems. The ENIAC (1945) was the first general-purpose electronic computer; the transistor (1947), integrated circuit (1958), and microprocessor (1971) drove exponential miniaturisation captured by Moore's Law. The personal-computer era (1977 onward), the internet (1980s research, 1990s commercial), and the smartphone (2007 iPhone) each shifted computing's role in society dramatically. The current era is defined by cloud computing (utility-priced compute and storage), mobile-first design, and generative AI.

Canada plays a distinctive role: Geoffrey Hinton, Yoshua Bengio, and Yann LeCun (the "godfathers of deep learning") all studied or worked in Canadian universities, and Toronto and Montreal remain global hubs for AI research. The Vector Institute (Toronto), Mila (Montreal), and Amii (Edmonton) are world-class research centres. Major Canadian post-secondary computer-science programs include Waterloo, Toronto, McGill, UBC, McMaster, Western, Queen's, Carleton, Ottawa, Simon Fraser, Alberta, and the polytechnic colleges (Sheridan, Seneca, BCIT) for applied programs.

Career paths in computing are now extremely diverse. Software developer remains the most common role; specialisations include front-end (browser UI), back-end (server APIs), full-stack (both), mobile, game development, and embedded systems. Data scientist and machine-learning engineer roles apply statistics and ML to business problems. Cybersecurity analysts defend systems against attack. DevOps and site reliability engineers run infrastructure. Cloud architects, database administrators, QA engineers, technical writers, product managers, and UX designers round out a typical software organisation. Many of these careers do not require a four-year university degree — community college programs and self-directed pathways are increasingly viable.

  • Turing machine — abstract model of computation that defines what is computable.
  • Moore's Law — observation that transistor counts double roughly every 2 years.
  • Cloud computing — on-demand delivery of compute and storage over the internet.
  • DevOps — culture and tools that combine development and operations.
  • SRE (Site Reliability Engineering) — software-engineering approach to operations.
  • FAANG / MAANG — large US tech firms; Canadian equivalents include Shopify, OpenText, Constellation, BlackBerry.
  • Co-op / internship — paid work term integrated with study (Waterloo's program is world-famous).
  • Open source contribution — public portfolio that demonstrates skills to employers.

Worked example — a career-planning checklist for an Ontario Grade 12 student

  • Foundations — solid grasp of one mainstream language (Python or Java), data structures, and Git. ICS4U covers all three.
  • Math — calculus and discrete math help in CS programs; statistics is essential for data roles.
  • Public portfolio — a GitHub profile with three or four substantive projects beats a résumé alone.
  • Choose an entry point — university (broad theory + research access), polytechnic college (faster, applied), bootcamp (intensive, networking-oriented), self-study (cheapest but requires discipline).
  • Apply early — Waterloo, Toronto, and McGill CS admissions are highly competitive; OUAC applications open in fall of Grade 12.
  • Co-op — programs like Waterloo CS or Toronto's PEY interleave paid work terms with study; many graduates have 2 years of industry experience at convocation.
  • Stay learning — technology shifts every 5-10 years; the durable skills are reading code, debugging, communicating, and learning fast.
  • Believing CS is just "coding" — the field is much broader (architecture, security, theory, HCI, applied math).
  • Picking a university by ranking alone — fit, co-op opportunities, and city matter more for outcomes.
  • Treating new tools as "fashions" to ignore — yesterday's fashion (Git, the cloud, ML) is today's foundation.
  • Underestimating soft skills — code review, technical writing, and collaboration are graded on the job.
  • Assuming the AI revolution makes programmers obsolete — it raises the floor of productivity but still rewards rigour and judgement.

Curriculum expectation: Strand D, Expectation D3.1 — "describe postsecondary education and training opportunities and career prospects related to computer studies".
