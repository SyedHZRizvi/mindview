1.1 — Variables, Data Types, and Expressions
Primitive types (int, float, str, bool); variable assignment; type conversion; operator precedence; basic expression evaluation
# Variable assignment and type conversion name = "Ada" age = 17 gpa = 3.85 is_senior = True # Build a record string with explicit conversions record = name + " (" + str(age) + "): " + str(gpa) print(record) # Ada (17): 3.85
📖 Lesson Notes — Ontario Curriculum

Every value a program manipulates has a data type, and every type carries with it a set of legal operations. Choosing the right primitive type is the first decision a programmer makes because it controls memory layout, the range of representable values, and which operators are valid. Confusing types is the source of a surprising number of bugs: "7" + "8" produces the string "78" in Python and Java, but 7 + 8 produces the integer 15. Grade 12 students must learn to reason about types before they reason about logic.

Python uses dynamic typing: a variable name is a label that can be rebound to any object. By contrast, Java uses static typing, where the compiler verifies the type at every assignment. In both languages, the value stored in memory has a fixed type at any given moment. Type conversion (also called casting) is the act of producing a new value of a different type — for example, str(age) creates a string version of the integer age without altering the original. Implicit conversion (3 + 4.0 becoming 7.0) is called type coercion and is the most common source of silent floating-point error.

Expressions are built from operators and operands, and the order in which they evaluate is governed by operator precedence. Multiplication binds tighter than addition; exponentiation in Python (**) binds tighter than unary minus on the left but not on the right; comparison binds looser than arithmetic. When in doubt, add parentheses — they cost nothing at runtime and prevent precedence bugs.

  • Primitive type — a built-in type that is not a collection (int, float, bool, str).
  • Literal — a value written directly in source code, e.g. 17 or "Ada".
  • Identifier — the name of a variable, function, or class.
  • Casting / type conversion — explicit conversion from one type to another, e.g. int("12").
  • Operator precedence — rules that determine the order in which sub-expressions evaluate.
  • Floating-point error — the rounding caused by binary representation of decimal fractions.

Worked example — precedence and floating-point

>>> 2 + 3 * 4 ** 2     # ** first → 16, then * → 48, then + → 50
50
>>> (2 + 3) * 4 ** 2   # () forces 5 first, ** still binds tighter than *
80
>>> 0.1 + 0.2          # binary cannot represent 0.1 exactly
0.30000000000000004
>>> round(0.1 + 0.2, 2)
0.3

The exponent 4 ** 2 evaluates first because ** has the highest precedence among arithmetic operators. Floating-point arithmetic in IEEE-754 cannot represent 0.1 exactly, so equality comparisons on floats should use a tolerance (abs(a - b) < 1e-9) rather than ==. Each operation here is O(1), so the running time of the expression is constant regardless of operand magnitude.

  • Forgetting to convert input() from string to number: input() + 1 raises TypeError.
  • Comparing floats with == — always use a tolerance.
  • Using = when you mean ==, or vice versa.
  • Assuming integer division: in Python 3, 5 / 2 is 2.5, not 2. Use // for floor division.
  • Overwriting a built-in name like list or str with your own variable.

Curriculum expectation: Strand A, Expectation A1.1 — "demonstrate the use of different data types … in computer programs (e.g., integer, floating-point, character/string, Boolean)".

1.2 — Input, Output, and String Formatting
input(), print(), f-strings, escape sequences, integer vs. float input parsing
# Build an interactive area calculator length = float(input("Length (m): ")) width = float(input("Width (m): ")) area = length * width print(f"Area = {area:.2f} m²")
📖 Lesson Notes — Ontario Curriculum

A program that cannot communicate with the outside world is not very useful. Input and output (I/O) are the bridges between the program's internal state and the user or another system. In Python, input() always returns a str, regardless of what the user types — even "42" arrives as four characters, not a number. The programmer must explicitly parse it with int() or float() before performing arithmetic. Forgetting this conversion is the single most common bug in introductory programs.

Output formatting matters because raw values rarely look the way humans want to read them. A bank balance of 1234.5 should display as $1,234.50 with two decimal places and a thousands separator. Python's f-strings (formatted string literals, introduced in Python 3.6) embed expressions directly inside string literals with a format specifier after a colon: f"{value:,.2f}" means "thousands separator, two decimal places, fixed-point notation". F-strings are faster, easier to read, and less error-prone than the older % operator or the .format() method.

Escape sequences let strings contain characters that would otherwise have special meaning. \n produces a newline, \t produces a tab, \\ produces a literal backslash, and \" escapes a double-quote inside a double-quoted string. Raw strings, prefixed with r, disable escapes — they are handy for Windows file paths and regular expressions.

  • Standard input (stdin) — the default source of input, usually the keyboard.
  • Standard output (stdout) — the default destination of output, usually the terminal.
  • f-string — an expression-embedded literal of the form f"…{expr:spec}…".
  • Format specifier — the part after the colon (.2f, >10, 05d) that controls width, precision, alignment.
  • Escape sequence — a two-character code beginning with \ that represents a non-printable character.
  • Concatenation — joining strings end-to-end with +.

Worked example — receipt formatting

item  = input("Item: ")          # str
qty   = int(input("Qty: "))      # int — parse!
price = float(input("Price: "))  # float — parse!

subtotal = qty * price
tax      = subtotal * 0.13       # Ontario HST
total    = subtotal + tax

print(f"{'Item':<15}{'Qty':>5}{'Price':>10}")
print(f"{item:<15}{qty:>5}{price:>10.2f}")
print(f"{'Subtotal':<20}{subtotal:>10.2f}")
print(f"{'HST (13%)':<20}{tax:>10.2f}")
print(f"{'TOTAL':<20}{total:>10.2f}")

The <15 specifier left-aligns to width 15; >10.2f right-aligns a float in width 10 with two decimals. Each print is an O(1) I/O operation. The arithmetic itself is O(1); the runtime is dominated by I/O latency, not computation.

  • Forgetting to wrap input() with int() or float() — leads to TypeError when arithmetic is attempted.
  • Calling int("3.5") — raises ValueError. Use float() first, then int() if truncation is wanted.
  • Using print(x, y) (which auto-inserts a space) when concatenation was intended.
  • Forgetting the f prefix: "{x}" is the literal string {x}, not the value of x.
  • Mixing tabs and spaces in formatted output — terminals may render them differently.

Curriculum expectation: Strand A, Expectation A1.2 — "use methods to manage data input and output in programs (e.g., format output to be displayed; validate input)".

1.3 — Selection: if/elif/else and Boolean Logic
Comparison operators, logical and/or/not, truth tables, short-circuit evaluation, the dangling-else problem
# Letter-grade classifier (Ontario Achievement Chart) def letter_grade(percent): if percent >= 80: return "Level 4" elif percent >= 70: return "Level 3" elif percent >= 60: return "Level 2" elif percent >= 50: return "Level 1" else: return "R"
📖 Lesson Notes — Ontario Curriculum

Selection — choosing which branch of code to run — is one of the three pillars of structured programming alongside sequence and iteration. The Boolean expression in an if evaluates to either True or False, and the interpreter takes the corresponding branch. Cascaded if/elif/else chains are mutually exclusive: at most one branch executes, and they are evaluated top-to-bottom. The order of conditions is therefore semantically meaningful, not just stylistic. Writing the most restrictive condition first is a frequent source of bugs (e.g., if percent >= 50 before if percent >= 80 would classify every passing grade as Level 1).

Boolean logic is built from the three connectives and, or, and not. A truth table enumerates all input combinations. Python (and Java) use short-circuit evaluation: in A and B, if A is false, B is never evaluated; in A or B, if A is true, B is never evaluated. This is more than an optimisation — it lets programmers write guards like if x != 0 and 1/x > 0.5 without raising a division-by-zero exception when x is zero.

De Morgan's laws are essential for refactoring conditionals: not (A and B) is equivalent to (not A) or (not B), and not (A or B) is equivalent to (not A) and (not B). Students should be able to apply these mechanically to simplify nested negations.

  • Boolean expression — any expression that evaluates to True or False.
  • Comparison operator — ==, !=, <, <=, >, >=.
  • Logical operator — and, or, not (in Java: &&, ||, !).
  • Short-circuit evaluation — second operand is skipped when the first determines the result.
  • Truthy / Falsy — non-Boolean values evaluated in a Boolean context (0, "", [], None are falsy).
  • Dangling-else — ambiguity in nested ifs without braces, resolved in Python by indentation.

Worked example — input validation with short-circuit logic

def safe_divide(a_str, b_str):
    if not a_str.lstrip('-').isdigit() or not b_str.lstrip('-').isdigit():
        return "Inputs must be integers"
    a, b = int(a_str), int(b_str)
    if b == 0:
        return "Cannot divide by zero"
    return a / b

print(safe_divide("12", "3"))   # 4.0
print(safe_divide("12", "0"))   # Cannot divide by zero
print(safe_divide("12", "x"))   # Inputs must be integers

The or in the first check short-circuits: if a_str is non-numeric, b_str is never tested. This protects against the ValueError that would arise from int("x"). The whole function runs in O(n) where n is the length of the input string (because of isdigit); the arithmetic itself is O(1).

  • Writing if x = 5 instead of if x == 5 (a syntax error in Python, a logic bug in C/Java).
  • Putting the most general condition first in an if/elif chain, making later branches unreachable.
  • Chaining comparisons in Java the way Python allows: 1 < x < 10 works in Python; in Java write 1 < x && x < 10.
  • Forgetting that and/or return one of the operands, not necessarily True/False: 0 or "hello" returns "hello".
  • Negating a compound condition without De Morgan's laws — easy to invert the wrong operator.

Curriculum expectation: Strand A, Expectation A2.1 — "use control structures (i.e., sequence, selection, iteration) to write modular code".

1.4 — Iteration: for, while, and Loop Tracing
Counted loops with range(), conditional loops, accumulator pattern, sentinel-controlled loops, off-by-one errors
# Sum of numbers 1..n using accumulator pattern def sum_to(n): total = 0 for i in range(1, n + 1): total += i return total print(sum_to(100)) # 5050 (Gauss)
📖 Lesson Notes — Ontario Curriculum

Iteration — repeating a block of code — is what makes a computer dramatically more powerful than a calculator. A for loop iterates over a known sequence (a list, a range, a string, a file), and the loop variable takes on each element in turn. A while loop continues as long as a Boolean condition holds, and is used when the number of iterations is not known in advance (e.g., reading until end-of-file, or polling for user input). Choosing between them is a design decision: for when you can describe what to iterate over; while when you need to describe when to stop.

The accumulator pattern initialises a result variable outside the loop, updates it inside the loop, and returns it after the loop. The accumulator can be a number (sum, product, count), a Boolean (any/all), a list (filter, map), or a dictionary (frequency count). Recognising this pattern lets students translate plain-English questions ("how many students passed?") into code mechanically.

The sentinel-controlled loop is a while idiom where a special "stop" value (like "quit", -1, or empty string) ends the loop. The sentinel must be a value that could not appear as legitimate data; choosing it carelessly is a famous source of bugs.

  • Loop body — the indented block that repeats.
  • Loop variable / index — the variable that changes each iteration.
  • Accumulator — a variable that builds up a result across iterations.
  • Sentinel — a special input value that signals the end of input.
  • Off-by-one error — looping one too few or one too many times, often from range(n) vs range(n+1).
  • Infinite loop — a while whose condition never becomes false.

Worked example — tracing a nested loop

def count_pairs_summing_to(target, nums):
    count = 0
    for i in range(len(nums)):
        for j in range(i + 1, len(nums)):  # j > i avoids dupes
            if nums[i] + nums[j] == target:
                count += 1
    return count

print(count_pairs_summing_to(10, [1, 9, 4, 6, 5, 5]))  # 3

Pairs that sum to 10: (1,9), (4,6), (5,5). The outer loop runs n times; the inner loop runs n-i-1 times for iteration i. Total comparisons = n(n-1)/2, so the time complexity is O(n²). Memory is O(1) — only a counter is stored. A faster O(n) algorithm uses a hash set to remember target - nums[i] values; this trade-off (time vs space) appears repeatedly in Unit 4.

Worked example — desk-checking a while loop

n = 12
steps = 0
while n > 1:
    if n % 2 == 0:
        n = n // 2
    else:
        n = 3*n + 1
    steps += 1
print(steps)  # 9 — the Collatz sequence

Trace: 12 → 6 → 3 → 10 → 5 → 16 → 8 → 4 → 2 → 1. Nine iterations. Whether this loop terminates for every positive n is the famous unsolved Collatz conjecture — a reminder that loop termination is not always trivial to prove.

  • Off-by-one: range(1, n) goes 1 to n-1, not 1 to n. Use range(1, n+1) to include n.
  • Modifying a list while iterating over it — leads to skipped or duplicated elements.
  • Infinite loops because the loop variable is never updated inside while.
  • Initialising an accumulator inside the loop instead of outside — resets it every iteration.
  • Using break in deeply nested loops without a clear exit strategy.

Curriculum expectation: Strand A, Expectation A2.1 — "use control structures (i.e., sequence, selection, iteration) to write modular code".

1.5 — Program Tracing and Desk Checks
Variable trace tables, dry runs, predicting output, the IPO (Input-Process-Output) chart
📖 Lesson Notes — Ontario Curriculum

Tracing — manually simulating the execution of a program with pen and paper — is one of the most important diagnostic skills a programmer develops. A trace table has one column per variable plus an "output" column, and one row per statement executed. By filling it in step-by-step, the student verifies that what the code actually does matches what they think it does. The technique is invaluable on tests, in code reviews, and during debugging, because it forces explicit reasoning about every state change.

The IPO chart (Input-Process-Output) is a higher-level planning tool used before coding. It lists what data enters the program, what transformations are applied, and what data leaves. IPO charts feed into structure charts, which decompose the "Process" box into a tree of sub-functions. Together they constitute the documentation a maintainer needs to understand a program's purpose without reading every line.

Tracing also reveals the difference between three classes of error: a syntax error prevents the program from running; a runtime error aborts the program mid-execution (e.g., divide-by-zero, index out of range); and a logic error lets the program complete but produces the wrong answer. Logic errors are the hardest to find precisely because the program "works" — tracing exposes them by comparing predicted output against actual output.

  • Trace table — a table tracking variable values line-by-line through execution.
  • Dry run — executing a program on paper without a computer.
  • IPO chart — a 3-column diagram listing Input, Process, Output.
  • Structure chart — a hierarchical diagram showing how modules call one another.
  • Syntax error — code that violates language grammar; caught by the parser.
  • Runtime error — an exception raised during execution.
  • Logic error — the program runs but produces incorrect output.
  • Invariant — a condition that holds true at a specific point in every iteration.

Worked example — trace this loop

def mystery(n):
    a, b = 0, 1
    while a < n:
        a, b = b, a + b
    return a

print(mystery(20))

Trace table:

iter | a  | b  | a < 20?
-----+----+----+--------
 0   | 0  | 1  | True
 1   | 1  | 1  | True
 2   | 1  | 2  | True
 3   | 2  | 3  | True
 4   | 3  | 5  | True
 5   | 5  | 8  | True
 6   | 8  | 13 | True
 7   | 13 | 21 | True
 8   | 21 | 34 | False → exit
return 21

The function returns the smallest Fibonacci number ≥ n. The loop invariant is "a and b are consecutive Fibonacci numbers". Time complexity is O(log φ · n) ≈ O(log n) because Fibonacci numbers grow exponentially. Memory is O(1).

  • Updating two variables sequentially when simultaneous update is required: a = b; b = a + b is not the same as a, b = b, a + b.
  • Forgetting to record output rows in the trace table — leads to missed off-by-one errors.
  • Tracing only a couple of iterations and assuming the pattern continues.
  • Confusing pre-condition (checked before the loop body) with post-condition (after).
  • Building an IPO chart with "process" steps that are vague verbs like "do the math" — every step should be testable.

Curriculum expectation: Strand A, Expectation A4.1 — "trace code, comments, and program output to identify the relationships among different sections of a program".
