2.1 — 1-D Lists (Arrays): Indexing, Slicing, Mutation
List literal syntax, zero-based indexing, negative indices, slice notation, common methods (.append, .pop, .insert, .remove)
scores = [82, 76, 91, 68, 88] print(scores[0]) # 82 (first) print(scores[-1]) # 88 (last) print(scores[1:4]) # [76, 91, 68] scores.append(95) # mutate in place avg = sum(scores) / len(scores)
📖 Lesson Notes — Ontario Curriculum

A list (in many languages called an array) is the single most useful data structure in computing. It stores an ordered, zero-indexed collection of elements in contiguous (or logically contiguous) memory, supporting O(1) random access via an integer index. Python lists are mutable and heterogeneous (they may hold mixed types), unlike Java arrays which are fixed-size and homogeneous. Python's list is implemented as a dynamic array of object references — appending amortises to O(1), but inserting at the front (insert(0, x)) is O(n) because every element must shift right.

Indexing in Python starts at 0 and runs to len(lst) - 1; negative indices count from the end (lst[-1] is the last element). Slicing with lst[start:stop:step] returns a new list — a critical fact, because modifying the slice does not affect the original. The half-open interval convention (start inclusive, stop exclusive) means lst[1:4] contains exactly 3 elements, not 4. This eliminates many off-by-one bugs once you internalise it: the length of a slice is always stop - start.

Mutability has consequences. Assigning b = a creates an alias, not a copy — both names refer to the same underlying list. b.append(7) changes what a sees. To make a shallow copy, use a[:], list(a), or a.copy(). For nested lists, use copy.deepcopy.

  • Index — a zero-based integer position into a list.
  • Slicing — extracting a sub-range with lst[start:stop:step].
  • Mutable — can be changed in place after creation (lists, dicts, sets).
  • Immutable — cannot be changed in place (strings, tuples, ints, floats).
  • Aliasing — two names referring to the same object.
  • Shallow copy — copies the top-level list but shares nested objects.
  • Amortised complexity — average cost per operation over a sequence of operations.

Worked example — building a histogram

def histogram(scores, bin_size=10):
    bins = [0] * (100 // bin_size + 1)   # 11 bins for 0-100
    for s in scores:
        bins[s // bin_size] += 1
    return bins

print(histogram([55, 62, 68, 71, 73, 84, 91, 91, 99]))
# [0, 0, 0, 0, 0, 1, 2, 2, 1, 2, 1]

Constructing the bins list is O(k) where k is the number of bins; the loop runs through n scores doing O(1) work each, so the total complexity is O(n + k). This is asymptotically optimal for counting because every score must be examined at least once. The space cost is O(k).

  • Index out of range: accessing lst[len(lst)] raises IndexError. Last valid index is len(lst) - 1.
  • Aliasing bug: b = a then thinking b is a copy. Use b = a[:] or b = list(a).
  • Multiplying a list of lists: [[0]*3]*3 creates three references to the same inner list. Use a list comprehension instead.
  • Calling lst.append expecting a new list back — it returns None.
  • Repeatedly inserting at the front when the desired order can be built by appending and reversing once at the end.

2.2 — 2-D Lists / Multi-Dimensional Arrays
Lists of lists, row/column indexing, nested loops to traverse a grid, building a tic-tac-toe or seating chart
# 3 x 3 game board (None = empty) board = [ [None, "X", None], [None, "O", None], ["X", None, None] ] for row in board: for cell in row: print(cell if cell else ".", end=" ") print()
📖 Lesson Notes — Ontario Curriculum

Two-dimensional data — grids, matrices, images, game boards, spreadsheets — show up everywhere in computing. In Python the natural representation is a list of lists: the outer list holds rows, each row is itself a list of cells. To address cell (r, c) we write grid[r][c]: the first index selects the row, the second selects the column. This convention (row-major) matches mathematical matrix notation and how C/Java store 2-D arrays in memory.

Traversing a 2-D structure almost always uses nested loops: the outer loop walks rows, the inner loop walks columns. The order matters for performance in compiled languages because of CPU cache lines, but at the Python level it changes only the order in which cells are visited. The total work for a full traversal of an r × c grid is O(r·c), which is linear in the number of cells.

A frequent pitfall is creating an "empty" 2-D grid by multiplying a row: [[0]*c]*r looks reasonable but produces a list where every row is the same object. Writing to grid[0][1] also writes to grid[1][1], grid[2][1], and so on. The correct idiom is a nested list comprehension: [[0]*c for _ in range(r)], which evaluates the inner expression independently for each row.

  • 2-D list / matrix — a list whose elements are themselves lists of equal length.
  • Row-major order — storing elements row-by-row in memory; the natural Python layout.
  • Nested loop — a loop inside another loop, giving multiplicative iteration counts.
  • Dimensions — the number of rows and columns, often written r × c.
  • Sparse vs dense — whether most cells hold meaningful data (dense) or default values (sparse).

Worked example — Conway's Life one-step update

def neighbour_count(grid, r, c):
    rows, cols = len(grid), len(grid[0])
    count = 0
    for dr in (-1, 0, 1):
        for dc in (-1, 0, 1):
            if dr == 0 and dc == 0:
                continue
            nr, nc = r + dr, c + dc
            if 0 <= nr < rows and 0 <= nc < cols:
                count += grid[nr][nc]
    return count

def step(grid):
    rows, cols = len(grid), len(grid[0])
    new = [[0]*cols for _ in range(rows)]   # independent rows!
    for r in range(rows):
        for c in range(cols):
            n = neighbour_count(grid, r, c)
            if grid[r][c] == 1 and n in (2, 3):
                new[r][c] = 1
            elif grid[r][c] == 0 and n == 3:
                new[r][c] = 1
    return new

For each of the r·c cells we examine 8 neighbours, all O(1) work, so a single step costs O(r·c). Bounds checking (0 <= nr < rows) handles edge cells without special cases. The new grid is built fresh to avoid reading half-updated state.

  • Initialising with [[0]*c]*r — every row aliases the same list. Use a comprehension.
  • Swapping r and c in the index: grid[c][r] when you meant grid[r][c].
  • Updating the grid in place while reading it — neighbouring cells see the new state, not the old.
  • Forgetting to bounds-check at edges, producing IndexError or wraparound from negative indices.
  • Assuming all rows have the same length — they don't, unless you enforce it.

2.3 — Dictionaries (Maps) and Records
Key-value pairs, lookup vs. linear search, dictionaries as records, nested dictionaries
student = { "name": "Hopper", "id": 100451, "grades": {"ICS4U": 94, "MCV4U": 88} } print(student["grades"]["ICS4U"]) # 94
📖 Lesson Notes — Ontario Curriculum

A dictionary (also called a map, hash table, or associative array) maps keys to values. Where a list is indexed by integer position, a dictionary is indexed by arbitrary immutable values — most commonly strings. Internally, Python uses an open-addressed hash table, which gives expected O(1) lookup, insertion, and deletion. Compare this with a linear search through a list of n records (O(n)): on a roster of 10,000 students, a dictionary lookup is roughly 10,000× faster.

Keys must be hashable, which in practice means immutable: strings, numbers, and tuples-of-immutables work; lists and dicts do not. The hash function maps each key to a bucket; collisions are resolved by probing. When the table fills past a load factor (~⅔), it is resized — an amortised cost that does not affect the O(1) expected bound.

Dictionaries shine as records: bundling related fields under named keys (student["name"], student["gpa"]) makes code self-documenting compared with parallel lists or magic tuple positions. They are also the workhorse for counting (frequency tables), grouping (bucketing by key), memoisation (caching computed results), and graph adjacency (mapping vertex → list of neighbours).

  • Key-value pair — the unit of storage in a dict.
  • Hash function — maps a key to an integer bucket index.
  • Hashable — has a stable hash; in Python, immutable types are hashable.
  • Collision — two keys hash to the same bucket.
  • Load factor — ratio of stored entries to bucket count.
  • Memoisation — caching the result of an expensive function in a dict.

Worked example — word-frequency counter

def word_frequencies(text):
    freq = {}
    for word in text.lower().split():
        word = word.strip(".,!?;:\"'")
        if word:
            freq[word] = freq.get(word, 0) + 1
    return freq

text = "To be, or not to be: that is the question."
print(word_frequencies(text))
# {'to': 2, 'be': 2, 'or': 1, 'not': 1, 'that': 1, 'is': 1, 'the': 1, 'question': 1}

The .get(word, 0) idiom returns 0 when the key is missing, avoiding a KeyError. The total cost is O(n) where n is the number of words: each iteration does O(1) hash work, and the final dict has at most n entries. For comparison, doing the same job with a list of (word, count) tuples and a linear search would be O(n²) — quadratically slower as the text grows.

Worked example — memoised Fibonacci

cache = {}
def fib(n):
    if n in cache:
        return cache[n]
    if n < 2:
        return n
    cache[n] = fib(n-1) + fib(n-2)
    return cache[n]

print(fib(50))  # 12586269025 — instant with memoisation

Without the cache, naive recursive Fibonacci is O(φⁿ) ≈ O(1.618ⁿ); with memoisation each n is computed once and stored, dropping the cost to O(n) time and O(n) space. This is the same principle that powers dynamic programming.

  • Using a list as a dict key — raises TypeError: unhashable type. Convert to tuple first.
  • Accessing a missing key with d[k] — raises KeyError. Use d.get(k, default) or if k in d.
  • Iterating over d assuming it yields values — it yields keys. Use d.values() or d.items().
  • Mutating a dict while iterating over it — raises RuntimeError. Iterate over list(d.keys()) instead.
  • Assuming order: insertion order is preserved since Python 3.7, but never rely on sorted order.

Curriculum expectation: Strand A, Expectation A1.3 — "demonstrate the ability to use one-dimensional arrays of compound data types (e.g., records) in computer programs".

2.4 — Strings as Sequences
Strings are immutable sequences; common methods (.split, .join, .strip, .lower, .replace, .find); slicing and reversal
📖 Lesson Notes — Ontario Curriculum

A string is an immutable sequence of characters. Many of the operations that work on lists — indexing, slicing, iteration, len, in — also work on strings. The crucial difference is mutability: you cannot write s[0] = 'X' because strings cannot be modified in place. Every method that "changes" a string actually returns a new string. For long-running text processing this matters: building a result by repeated concatenation (result = result + char) is O(n²) because each + copies the entire accumulated string. The idiomatic fix is to collect parts in a list and call "".join(parts) at the end, which is O(n).

Python strings are Unicode (UTF-8 internally on most platforms); each character is a code point, not a byte. This means len("café") is 4, not 5 — but writing it to a file in UTF-8 produces 5 bytes because é takes two bytes. Awareness of the text/bytes distinction prevents the classic "works on my machine until the input contains an accent" bug.

String methods form a small standard library every Grade 12 student should memorise: .split(sep) tokenises into a list, sep.join(lst) is its inverse, .strip() removes leading/trailing whitespace, .lower() and .upper() change case, .replace(old, new) substitutes, .find(sub) returns the first index or -1, .startswith/.endswith are useful for parsing.

  • Immutable — value cannot be modified after creation.
  • Code point — a Unicode integer identifying one character.
  • Encoding — the byte sequence used to store a code point (UTF-8, UTF-16, ASCII).
  • Substring — a contiguous slice of a string.
  • Tokenisation — splitting a string into meaningful units (words, fields).
  • Palindrome — a string that reads the same forwards and backwards.

Worked example — palindrome check and CSV parse

def is_palindrome(s):
    s = ''.join(c.lower() for c in s if c.isalnum())
    return s == s[::-1]   # [::-1] reverses

print(is_palindrome("A man, a plan, a canal: Panama"))  # True
print(is_palindrome("ICS4U"))                            # False

def parse_csv_row(row):
    return [field.strip() for field in row.split(',')]

print(parse_csv_row(" Ada , 17, 3.85 "))  # ['Ada', '17', '3.85']

The palindrome routine first filters out non-alphanumeric characters and lowercases each one — O(n). The slice s[::-1] creates a reversed copy in O(n), and the equality check is O(n). Total: O(n) time, O(n) space. The CSV parser uses .split(',') followed by .strip() on each field; this is also O(n) but ignores the subtleties of quoted fields — the csv module handles those properly.

  • Trying to assign to an index: s[0] = 'X' raises TypeError.
  • Building a result with += in a loop — O(n²). Use "".join(list_of_parts).
  • Confusing .find (returns -1 on miss) with .index (raises ValueError).
  • Treating str.split() with no argument as the same as .split(' ') — they differ on runs of whitespace.
  • Forgetting that "7" < "10" is True (lexicographic), not False (numeric).

Curriculum expectation: Strand A, Expectation A1.1 — "demonstrate the use of different data types (e.g., integer, floating-point, character/string, Boolean) in computer programs".

2.5 — File I/O: Reading and Writing Text Files
open(), context managers (with statement), reading lines, writing CSV-like data
# Read a roster file, output the average mark with open("roster.csv") as f: total, count = 0, 0 for line in f: name, mark = line.strip().split(",") total += int(mark) count += 1 print(f"Class average: {total/count:.1f}")
📖 Lesson Notes — Ontario Curriculum

Programs need persistence — the ability to save data between runs. Files are the simplest form of persistence: a named, byte-addressed region on disk. Python's open(path, mode) returns a file object; mode is "r" for read, "w" for write (truncates existing content), "a" for append, and "b" appended for binary. A file object is iterable in line-by-line fashion, which means for line in f reads one line at a time without loading the entire file into memory — essential for large data sets.

The with statement is a context manager that guarantees the file is closed even if an exception is raised inside the block. Forgetting to close a file can leak operating-system handles and, on Windows, prevent other programs from accessing it. with is the modern, idiomatic way to handle files; bare open without close is a code-review red flag.

Text files store lines terminated by \n (Unix) or \r\n (Windows); Python's text-mode I/O normalises these to \n on read and the platform default on write. The CSV (comma-separated values) format is the simplest tabular interchange format: one row per line, fields separated by commas. The csv module in the standard library handles quoted fields, escaped commas, and varied dialects — students should prefer it over hand-rolled .split(',') once data gets messy.

  • File handle — the OS-level reference returned by open.
  • Mode — "r", "w", "a", optionally with "b" for binary or "+" for read/write.
  • Context manager — an object usable with with that defines setup/teardown.
  • Buffering — accumulating writes before flushing to disk for efficiency.
  • EOF (end-of-file) — signalled by an empty string from read or by the loop ending.
  • CSV — a plain-text tabular format.

Worked example — read, transform, write

import csv

def upgrade_marks(in_path, out_path, bonus=5):
    with open(in_path, newline='') as fin, \
         open(out_path, 'w', newline='') as fout:
        reader = csv.DictReader(fin)
        writer = csv.DictWriter(fout, fieldnames=reader.fieldnames)
        writer.writeheader()
        for row in reader:
            row['mark'] = min(100, int(row['mark']) + bonus)
            writer.writerow(row)

upgrade_marks('marks.csv', 'marks_curved.csv')

Two files are opened in a single with; both are guaranteed to close. The reader and writer stream line-by-line so memory use is O(1) regardless of file size. Total time is O(n) in the number of rows. The min(100, …) guard caps marks so a bonus never pushes a 98 to 103. Real Ontario reporting policies vary; this is just an illustration.

  • Opening a file in "w" mode "to check it exists" — it will be truncated to empty.
  • Forgetting with and not calling .close(), leaking handles.
  • Reading the whole file into memory with f.read() when streaming would suffice.
  • Forgetting that line from iteration includes the trailing \n; use line.rstrip("\n").
  • Hand-parsing CSV with split(',') when fields may contain commas inside quotes.
  • Hard-coding absolute paths that break on another machine — use relative paths or pathlib.

Curriculum expectation: Strand A, Expectation A1.2 — "use methods to manage data input and output in programs (e.g., to read from and write to a sequential file)".

2.6 — Classes and Objects (Introduction to OOP)
Defining a class, attributes, methods, the __init__ constructor, encapsulation; objects vs. records
class Student: def __init__(self, name, grade): self.name = name self.grade = grade def promote(self): self.grade += 1 def __str__(self): return f"{self.name} (Grade {self.grade})" s = Student("Turing", 11) s.promote() print(s) # Turing (Grade 12)
📖 Lesson Notes — Ontario Curriculum

Object-oriented programming (OOP) bundles related data and the operations on that data into a single unit called a class. A class is a blueprint; an instance (or object) is a concrete realisation in memory. The instance's attributes hold its data, and its methods are functions that operate on that data. The __init__ method (the constructor) runs automatically when a new instance is created, initialising attributes from the arguments passed.

OOP's main benefits are encapsulation (data and behaviour live together, so external code does not need to know representation details), abstraction (a clean interface hides complexity), and (once we add inheritance) reuse. A Student object knows its own name and grade and knows how to promote itself — calling code just writes s.promote() rather than manipulating fields directly. This makes the program easier to read, easier to change (the implementation can swap out without touching callers), and easier to test (each class can be unit-tested in isolation).

The self parameter is the explicit reference to the current instance. In Python you must list it; Java's this is implicit. The __str__ "dunder" method (double-underscore on both sides) defines how the object is converted to a string for print or str(). Providing a meaningful __str__ is a small effort that pays huge dividends during debugging.

  • Class — a blueprint defining attributes and methods.
  • Instance / object — a concrete value created from a class.
  • Attribute — a piece of data stored on an instance (or class).
  • Method — a function defined inside a class.
  • Constructor — the __init__ method that runs at creation.
  • Encapsulation — bundling state with the methods that change it.
  • Dunder method — a special method named with double underscores (e.g. __str__, __eq__).

Worked example — a BankAccount class

class BankAccount:
    def __init__(self, owner, opening_balance=0):
        self.owner = owner
        self._balance = opening_balance        # leading _ = "private"

    def deposit(self, amount):
        if amount <= 0:
            raise ValueError("amount must be positive")
        self._balance += amount

    def withdraw(self, amount):
        if amount > self._balance:
            raise ValueError("insufficient funds")
        self._balance -= amount

    def balance(self):
        return self._balance

    def __str__(self):
        return f"{self.owner}: ${self._balance:,.2f}"

a = BankAccount("Ada", 100)
a.deposit(50)
a.withdraw(30)
print(a)            # Ada: $120.00
print(a.balance())  # 120

Each operation runs in O(1) and uses O(1) extra memory per account. The leading underscore on _balance is a convention signalling "do not access from outside the class" — Python does not enforce it but reviewers do. Encapsulation here prevents callers from corrupting the balance directly; the only legal mutations go through deposit and withdraw, which validate their arguments.

  • Forgetting self as the first parameter of every instance method — produces confusing TypeErrors.
  • Defining attributes outside __init__ (as class variables) and then mutating them — they are shared across all instances.
  • Using a mutable default argument: def __init__(self, items=[]) — every instance shares the same list!
  • Returning self from a setter and then assigning obj = obj.set_x(5) — works, but confuses students about mutation vs assignment.
  • Comparing objects with == without overriding __eq__ — falls back to identity comparison.
